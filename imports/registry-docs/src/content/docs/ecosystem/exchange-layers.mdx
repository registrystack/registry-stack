Exchange layers connect institutions, while Registry Stack can sit behind those layers to expose
entity-shaped routes, registry metadata, and purpose-specific evidence responses.

## Ecosystem role

An exchange layer often owns network-level concerns: participant onboarding, message transport,
mutual trust, routing, addressing, and cross-institution policy.
In deployments where that layer already exists, Registry Stack does not need to replace it.

Registry Stack is complementary at the registry edge.
It describes the registry surface, exposes protected read routes, and certifies evidence responses
for callers that reach the registry through an approved institutional path.

## Layered pattern

Use a layered pattern when an exchange layer is deployed:

- The exchange layer handles institutional transport and trust where deployed.
- Registry Relay can sit behind the exchange layer for entity-shaped protected routes.
- Registry metadata describes fields, schemas, policies, and evidence offerings.
- Registry Witness can provide purpose-specific evidence responses forwarded by the exchange layer.

This means the exchange layer can keep doing network and participant governance work, while the
registry-facing service keeps field exposure, schema publication, evidence shape, and audit behavior
close to the authoritative source.

## Where overlap exists

Overlap appears around authorization, routing, audit, and API publication.
An exchange layer may already authenticate participants, authorize message flows, and log messages.
Registry Relay still needs route-level scopes, caller-aware metadata, field projection, and source
bindings because those controls are specific to the registry contract.

The same separation applies to evidence.
The exchange layer can forward a request and preserve institutional trust context.
Registry Witness evaluates the configured claim and returns the configured response shape.

## Standards fit

OpenAPI describes the HTTP route contract.
DCAT, BRegDCAT-AP, SHACL, JSON Schema, and ODRL describe registry metadata, data service shape, and
policy.
CCCEV and SD-JWT VC apply when the response is evidence-shaped or credential-shaped.
W3C Decentralized Identifiers (DID Core) apply only to the narrow DID surfaces documented by the
runtime services.
