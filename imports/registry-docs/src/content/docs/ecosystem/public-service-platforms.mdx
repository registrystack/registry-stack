Public service platforms coordinate service delivery, while Registry Stack describes,
protects, and proves registry facts that those platforms need during delivery.

## Ecosystem role

A public service platform often owns the user journey: forms, case queues, workflow state,
notifications, payments, appointments, and channel management.
It may also publish a service catalogue and integrate with one or more authoritative registries.

Registry Stack is complementary to that platform role.
Registry Manifest can publish service, requirement, evidence, policy, and registry metadata that a
platform can inspect before integration.
Registry Relay can expose scoped, read-only registry routes behind platform workflows.
Registry Witness can return configured evidence responses when a service needs proof instead of a
full record.

## Where overlap exists

The overlap is real around service catalogues, evidence requirements, eligibility checks, and API
facades.
A public service platform may already model a service, collect evidence, and call registries.
Registry Stack does not replace those workflow and channel responsibilities.
It narrows the registry-facing surface: which entity routes exist, which fields are exposed, which
policy applies, and which evidence response can be requested.

## Pattern

Use the public service platform as the service orchestration layer.
Use Registry Stack for registry metadata, purpose-bound consultation, evidence responses, and audit
records.

The boundary keeps program decisions in the service platform.
Registry Witness can certify that a configured claim evaluated against registry data, but the public
service platform decides what that result means for an application, entitlement, or case.

## Standards fit

CPSV-AP can describe services, competent authorities, channels, and requirements.
CCCEV can describe evidence types and claim results.
DCAT and BRegDCAT-AP can describe registries and data services.
ODRL can express policy metadata.
OpenAPI describes runtime HTTP contracts.

Registry Stack uses these standards as publication and integration surfaces.
It does not claim that a public service platform must adopt the whole stack or change its casework
model.
