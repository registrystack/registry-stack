Stay in these docs for cross-project questions: which project owns what, how the stack
fits together, what each standard claim means, what contracts are published.
Jump to the owning repository when you need implementation specifics, runtime configuration,
or service-internal contribution workflow.

Pick the path that matches your goal.

## I want to integrate with a registry

Start with [Commitments and capabilities](../what-we-built/) for the mental model of what the registry stack does.
Then read [Registry Relay](/projects/registry-relay/) to understand the entity-shaped API routes,
auth scopes, and the `Data-Purpose` header requirement.
Check the [contracts index](/reference/contracts/) for the OpenAPI documents and generated schema
artifacts you will integrate against.
If you need to call a Witness endpoint to verify eligibility or receive a credential, read
[Registry Witness](/projects/registry-witness/) next.
Run [your first call](../your-first-call/) to get a working request before writing integration
code.

## I'm evaluating standards conformance

Start with the [standards register](/reference/standards/) to see which standards each project
claims to emit, implement, or align with, and at what claim level.
Then use [Registry Atlas](/projects/registry-atlas/) to inspect a live or demo registry's
published artifacts in a browser: Atlas indexes the DCAT, CPSV-AP, CCCEV, SHACL, JSON Schema,
OpenAPI, and OGC Records artifacts and shows gaps.
For the per-standard explanation of how each artifact is generated, read
[Publishing pipeline](/explanation/publishing-pipeline/).
For the claim evaluation and credential issuance flow, read
[Evidence issuance](/explanation/evidence-issuance/).

## I'm contributing to a project repo

Read [Commitments and capabilities](../what-we-built/) for the boundary definitions: which project owns which
question.
Then go to the owning project page ([Registry Manifest](/projects/registry-manifest/),
[Registry Relay](/projects/registry-relay/),
[Registry Witness](/projects/registry-witness/),
[Registry Atlas](/projects/registry-atlas/),
[Registry Platform](/projects/registry-platform/), or
[Registry Lab](/projects/registry-lab/)) and follow the link to the source repository.
These docs own cross-project explanation.
Substantive code work, issues, and pull requests happen in the owning repos.

## I'm operating a deployment

Read [Registry Relay](/projects/registry-relay/) for source binding, auth configuration, scope
policy, and the `/metadata/*` publication surface.
Read [Registry Witness](/projects/registry-witness/) for claim configuration, disclosure policy,
and the OID4VCI issuance flow.
For a local end-to-end reference topology, read [Registry Lab](/projects/registry-lab/): it
wires three Relays, four Witnesses, Postgres, Zitadel, an OpenFn sidecar, and a static
metadata publisher into a single Docker Compose file with narrated smoke tests.
Registry Lab is not production deployment guidance, but the topology and configuration files are
a readable reference for how the services connect.
