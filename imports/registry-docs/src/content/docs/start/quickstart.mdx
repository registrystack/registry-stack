Pick the path that matches what you're trying to do.

## If you want to query a registry over HTTP

Start with [Registry Relay](../../projects/registry-relay/). Relay is the protected consultation gateway: it turns CSV, XLSX, Parquet, and PostgreSQL sources into authorized read-only APIs over entity routes like `household` or `individual`. Read its docs first, then check the [contracts index](../../reference/contracts/) for the specific API shapes you'll integrate against.

## If you're changing shared security or operational primitives

Start with [Registry Platform](../../projects/registry-platform/). Platform owns the reusable Rust crates for auth helpers, OIDC verification, audit envelopes, HTTP security, outbound HTTP policy, crypto, SD-JWT VC helpers, and test fixtures used by Relay and Witness.

## If you want to see the family running

Open the [Registry Lab tutorial](../../tutorials/first-run-with-registry-lab/). It's the smallest runnable demo: three Relay instances, three Witness instances, a static metadata publisher, and a demo client wired together in one Docker Compose topology.

## If you're integrating against the family

1. Read [what is the registry family](../what-is-the-registry-family/) for the one-page mental model.
2. Use the [project map](../../map/) to find the project that owns your specific question.
3. Check the [contracts index](../../reference/contracts/) for the APIs, schemas, and generated artifacts you'll talk to.

## If you're reviewing what a registry publishes

1. Read the [standards register](../../reference/standards/) to see which standards each project claims to emit, implement, or align with, and at what claim level.
2. Use [Registry Atlas](../../projects/registry-atlas/) to inspect a live registry's published artifacts in the browser.
3. Check [standards positioning](../../explanation/standards-positioning/) for the per-standard stance.

## If you're contributing

You're in the right place to orient. After that, the project pages link to each owning repo where issues, PRs, and contribution guidelines live. Registry Legend itself owns cross-project explanation; substantive code work happens in the owning repos.

## When to keep reading vs. jump to a repo

Stay in Registry Legend for cross-project questions: which project owns what, how the family fits together, what each standard claim means, what contracts are published. Jump to the owning repository when you need implementation specifics, runtime configuration, or service-internal contribution workflow.
