The registry stack is built for federated public administration. Each authority keeps custody
of its own registry data; the stack provides the API surface for lawful exchange between
authorities. It does not absorb data into a central system.

To make federation viable, the stack commits to three things.

## Three commitments

### Interoperability

Each registry publishes its catalog, schemas, services, and policies in standards-shaped form
(Data Catalog Vocabulary (DCAT), BRegDCAT-AP, Core Public Service Vocabulary Application
Profile (CPSV-AP), Core Criterion and Core Evidence Vocabulary (CCCEV), Shapes Constraint
Language (SHACL), JSON Schema, Open Digital Rights Language (ODRL), OpenAPI, SD-JWT VC).
Downstream systems integrate against stable contracts, not per-deployment ones.

### Safeguards

Authorized access only, with authentication by API key or OpenID Connect (OIDC). Scope-checked routes.
Per-claim disclosure control. No write paths. Audit envelopes for every request that touches
person-level data.

### Reviewability

Every commitment is backed by an artifact a reviewer can inspect: the published catalog, the
SHACL shapes, the policy documents, the audit envelope schema, the credential profile
configuration. Registry Atlas reads what a deployment actually emits, not what it claims to
emit.

## Three capabilities

The stack delivers these commitments through three capabilities: Publish, which turns a YAML
manifest into every standards artifact downstream systems need; Consult, which gives authorized
callers read-only, entity-shaped access to registry data over HTTP; and Prove, which evaluates
eligibility claims and issues verifiable credentials without exposing the records that back
them. Each capability maps to one or more projects.

### Publish

Publish answers the question: how does a registry declare what it contains, what
standards it follows, and what evidence it can provide, in a form every interested system can
read without talking to a running service?

#### Registry Manifest

[Registry Manifest](/projects/registry-manifest/) is a Rust library and CLI.
You write a single YAML metadata manifest describing a registry.
Registry Manifest reads that manifest and produces a static bundle of standards artifacts:

- catalog JSON
- Data Catalog Vocabulary (DCAT) and BRegDCAT-AP JSON-LD
- Core Public Service Vocabulary Application Profile (CPSV-AP) service catalogue JSON-LD
- Core Criterion and Core Evidence Vocabulary (CCCEV) requirement and evidence type metadata
- SHACL node shapes
- JSON Schemas and form JSON Schemas
- OGC API Records item bodies
- ODRL policy documents
- evidence-offering metadata

Registry Manifest requires no HTTP gateway, no database, and no credentials.
The output bundle can be served from any static host.
Because the bundle is generated from a single manifest, the published surface stays consistent
across all formats when the manifest changes.

#### Registry Atlas

[Registry Atlas](/projects/registry-atlas/) is a browser single-page application with a small
fetch proxy.
Atlas is the inspector for what was published: you point it at a published catalog or CPSV-AP
service catalogue endpoint and it indexes the DCAT, CPSV-AP, CCCEV, SHACL, JSON Schema,
OpenAPI, and OGC Records artifacts.
The primary view is service-first: public service, requirement, grouped evidence option,
provider, access service, and form.
Atlas enforces strict term matching; no fuzzy matching or synonym expansion.

Together, Manifest and Atlas close the Publish loop: Manifest generates the artifacts and Atlas
verifies that a running registry actually emits what it claims to emit, before you write
integration code.

### Consult

Consult answers the question: how do authorized systems query registry data over HTTP
without gaining database access or write paths?

#### Registry Relay

[Registry Relay](/projects/registry-relay/) is a config-driven Rust HTTP service.
You bind it to CSV, XLSX, Parquet, or PostgreSQL sources and configure entity shapes such as
`household` or `civil_person`.
Relay exposes those sources as protected, read-only, entity-shaped routes.
Authentication is API key or OIDC, backed by Registry Platform primitives.
Authorization is scope-checked per route: a credential that can verify evidence cannot read rows,
and a credential that can read rows cannot run aggregates.

Relay also publishes the same standards artifacts Registry Manifest renders, scoped to what
each caller is allowed to see, at runtime under `/metadata/*`.
This means a running Relay can serve its own DCAT catalog, CPSV-AP service catalogue, CCCEV
requirements, and evidence-offering metadata without a separate static publication step.

Relay is a restricted consultation gateway.
It has no write paths in v1 and it is not an open-data portal.

### Prove

Prove answers the question: how does a calling system learn whether a person qualifies
for a program, without exposing the records that determine that answer?

#### Registry Witness

[Registry Witness](/projects/registry-witness/) is a standalone Rust HTTP service.
You configure named eligibility claims, each with extraction rules or CEL expressions, disclosure
modes, and optional credential profiles.
When a caller submits a subject and a list of claims, Witness fetches the required registry facts
from Registry Relay (or an SP DCI-style source), evaluates each claim, and returns a
verification receipt.

The receipt format depends on what the caller requested:

- `application/vnd.registry-witness.claim-result+json` for the default structured result
- `application/ld+json; profile="cccev"` for a CCCEV JSON-LD receipt
- `application/dc+sd-jwt` for a Selective Disclosure JWT Verifiable Credential (SD-JWT VC)

When the caller requests an SD-JWT VC, Witness enforces the configured disclosure mode: `value`
reveals the underlying field, `predicate` reveals only whether the claim was satisfied, and
`redacted` omits the claim from the credential entirely.
Witness also supports OID4VCI issuance for wallet-compatible flows.

Witness is independent from Registry Relay application code.
Both services may depend on Registry Platform.
A verification receipt is the output of Witness.
Whether that receipt constitutes a programmatic eligibility decision is the caller's
responsibility, not a guarantee Witness provides.

## Shared infrastructure

[Registry Platform](/projects/registry-platform/) is a Rust crate workspace, not a running
service.
Registry Relay and Registry Witness both depend on it for:

- auth helpers (API key and OIDC verification)
- audit envelopes
- HTTP security middleware
- outbound HTTP policy
- Ed25519 JWK and DID helpers
- SD-JWT VC helpers
- shared test fixtures

Platform owns the behavior that must be consistent across all registry services.
Product-specific route behavior and policy decisions stay in the consuming services.

## Demo

[Registry Lab](/projects/registry-lab/) is a Docker Compose demo that wires Consult and
Prove together on a laptop.
It runs three Relay instances (civil registry on port `4311`, social protection on `4312`,
health on `4313`), four Witness instances (civil on `4321`, social protection on `4322`,
shared eligibility on `4323`, and an OpenFn-fed civil Witness on `4324`), an internal
`openfn-civil-sidecar` service that feeds the `4324` Witness, live Postgres and Zitadel,
and a static metadata publisher on port `4331`.
A separate host-side Witness on port `4325` (started outside compose by `just
citizen-oid4vci-*`) hosts the OID4VCI wallet flow.

Registry Atlas is run separately and is not part of the Compose topology at v0.1.

Lab is not production deployment guidance.
The demo does not represent a real integration with OpenCRVS, OpenSPP, DHIS2, OpenIMIS, MOSIP,
or any other production system.
