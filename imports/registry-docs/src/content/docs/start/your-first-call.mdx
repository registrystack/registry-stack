This tutorial makes two calls against a running Registry Lab: read one row from a Relay, then ask a
Notary to evaluate a claim about a person. Both calls use the demo fixtures that ship with Registry
Lab, so the results are reproducible on any machine with Docker. When you finish, you will have seen
the two moves every integration is built from, a scoped read and a disclosure-limited evidence
answer, and the guardrails that sit on each.

For the full topology, the smoke suite, credential issuance, and a cross-authority claim, continue
with [First run with Registry Lab](../../tutorials/first-run-with-registry-lab/) after this page.

**Prerequisites:** Docker (OrbStack or Docker Desktop) with Compose v2.20 or later, the
[`just`](https://github.com/casey/just) command runner, `uv`, `python3`, `openssl`, and `git` with
GitHub SSH access. The repository and its `vendor/*` submodules clone over SSH.
Estimated time: about 15 minutes the first time, most of it image builds; under 2 minutes on later
runs.

## Step 1: bring up Registry Lab

Clone the repository with its submodules and start the demo topology.

```sh
git clone --recurse-submodules git@github.com:jeremi/registry-lab.git
cd registry-lab
just generate
just build
just up
```

`just generate` writes `.env` with local demo credentials, generates the fixture data, prepares the
Postgres TLS material, and publishes the static metadata bundle. Do not hand-edit `.env`;
`just generate` rewrites it on every run.

`just up` starts the civil, social protection, and health Relay instances, the Notary instances,
Postgres, Zitadel, and the static metadata publisher. Wait until the services report healthy:

```sh
docker compose -f compose.yaml ps
```

Load the generated credentials into your shell before the calls below:

```sh
set -a
. ./.env
set +a
```

When you are done, stop the containers and remove the demo volumes:

```sh
just down
```

## Step 2: read one row from a Relay

The civil registry Relay listens on host port `4311` and serves civil person records from a CSV
fixture. Read one row:

```sh
curl -fsS \
  -H "Authorization: Bearer ${CIVIL_ROW_READER_RAW}" \
  -H "Data-Purpose: https://demo.example.gov/purpose/decentralized-evidence-demo" \
  -H "x-request-id: first-call-001" \
  "http://127.0.0.1:4311/v1/datasets/civil_registry/entities/civil_person/records?limit=1"
```

Expected response shape (the field values depend on the first fixture row):

```json
{
  "data": [
    {
      "national_id": "NID-1001",
      "given_name": "...",
      "surname": "...",
      "birth_date": "...",
      "civil_status": "...",
      "deceased": false,
      "district": "..."
    }
  ],
  "pagination": {
    "has_more": true,
    "next_cursor": "..."
  }
}
```

Two guardrails are worth noticing on this single call:

- The `Data-Purpose` header is required on the `civil_person` route. Omit it and the Relay returns
  `400`.
- The bearer token must carry the `civil_registry:rows` scope. A token holding only
  `civil_registry:evidence_verification` is rejected with `403` and the code `auth.scope_denied`.

Reads are scoped and purpose-tagged, and the Relay is read-only: there is no route that writes back
to the source.

## Step 3: evaluate a claim with a Notary

A Relay read returns data. A Notary answers a question about that data without returning the
underlying record. The civil Notary listens on host port `4321` and evaluates the `person-is-alive`
claim by fetching the `deceased` field from the civil Relay and applying the rule
`source.civil.deceased == false`.

Evaluate the claim for subject `NID-1001`:

```sh
curl -fsS -X POST \
  -H "Authorization: Bearer ${CIVIL_EVIDENCE_CLIENT_BEARER}" \
  -H "Content-Type: application/json" \
  -H "Accept: application/vnd.registry-notary.claim-result+json" \
  -H "Data-Purpose: https://demo.example.gov/purpose/decentralized-evidence-demo" \
  -H "x-request-id: first-call-002" \
  "http://127.0.0.1:4321/v1/evaluations" \
  --data '{
    "target": {"type": "Person", "identifiers": [{"scheme": "national_id", "value": "NID-1001"}]},
    "claims": ["person-is-alive"],
    "disclosure": "predicate",
    "format": "application/vnd.registry-notary.claim-result+json"
  }'
```

Expected response shape (trimmed to the fields you read first):

```json
{
  "results": [
    {
      "claim_id": "person-is-alive",
      "satisfied": true,
      "disclosure": "predicate",
      "format": "application/vnd.registry-notary.claim-result+json",
      "evaluation_id": "01HX...",
      "issued_at": "2026-06-07T12:00:00Z",
      "provenance": {
        "source_count": 1,
        "computed_by": "demo.registry-notary"
      }
    }
  ]
}
```

The result carries more fields than shown (`claim_version`, `subject_type`, `target_ref`,
`expires_at`, and `provenance.source_versions`); the ones above are enough to read the outcome.

What this call shows:

- `satisfied: true` means the rule evaluated to `true` for `NID-1001`.
- `disclosure: predicate` means the response states only whether the claim holds, not the `deceased`
  value it was derived from. That is the contrast with the Step 2 row read: the Notary confirmed a
  fact without disclosing the record behind it.
- The Notary fetched the source field itself, using its own configured credentials. The caller never
  reaches the civil Relay directly.
