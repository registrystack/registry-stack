The registry family is six projects that help you publish, consult, verify, and operate protected registry data: data that holds facts about people, households, businesses, or services and that authorized systems need to query without getting a copy of the underlying database.

## The six projects

### Registry Platform

Keep shared security and operational behavior consistent across registry services.

Registry Platform is a Rust crate workspace, not a running service. Relay and Witness consume it for auth helpers, OIDC verification, audit envelopes, HTTP security middleware, outbound HTTP policy, Ed25519 JWK and DID helpers, SD-JWT VC helpers, and test fixtures. Product-specific route behavior and policy decisions stay in the consuming services.

### Registry Manifest

Describe a registry once in YAML, then render every standards-facing artifact other systems need to find, understand, and connect to it, without having to stand up any runtime.

Registry Manifest is a Rust library and CLI. You point it at a metadata manifest and it produces a static bundle: catalog JSON, DCAT and BRegDCAT-AP JSON-LD, SHACL node shapes, JSON Schemas, OGC API Records item bodies, ODRL policy documents, and evidence-offering metadata. You can serve that bundle from any static host. No HTTP gateway, no database, no credentials.

### Registry Relay

Let authorized systems query a registry over HTTP without giving them database access or write paths.

Registry Relay is a config-driven Rust HTTP service. You bind it to CSV, XLSX, Parquet, or PostgreSQL sources, and it exposes them as protected, read-only, domain-oriented APIs. Routes are shaped by entities such as `household` or `individual`, not by table names. Auth is API key or OIDC, backed by Registry Platform primitives. Relay also publishes the same standards-facing artifacts Manifest renders, scoped to what each caller is allowed to see, at runtime under `/metadata/*`.

Relay is a restricted consultation gateway, not an open-data portal and not a write platform.

### Registry Witness

Prove a person qualifies for a program without sharing the records that prove it.

Registry Witness is a standalone Rust HTTP service. It evaluates configured eligibility claims against registry data (fetched over HTTP from Registry Relay or an SP DCI source) and produces a verification receipt. It can issue that receipt as an SD-JWT VC credential with selective disclosure, so the caller learns only what each claim authorized. Auth, audit envelopes, HTTP security, OIDC verification, and SD-JWT helper behavior are backed by Registry Platform primitives.

Witness is independent from Registry Relay application code: it doesn't import or link Relay libraries. Both services may depend on Registry Platform. Witness renders claim results as CCCEV JSON-LD when the caller asks for `application/ld+json; profile="cccev"`.

Witness produces a verification receipt. Treating it as a programmatic eligibility decision is the caller's choice, not a guarantee Witness makes.

### Registry Atlas

Explore what a registry publishes in your browser, and check whether it actually emits what it claims to emit, before you write integration code.

Registry Atlas is a browser single-page application with a small fetch proxy. You point it at a published catalog and it indexes the DCAT, SHACL, JSON Schemas, OpenAPI, and OGC Records artifacts, then lets you run strict capability queries against them. The v0.1 matcher is strict by design: no fuzzy matching, no synonym expansion, no AI inference. Every match needs an exact term correspondence.

### Registry Lab

Run the registry runtime services together on your laptop in one `docker compose up`.

Registry Lab is a Compose-based runnable demo. It wires three Relay instances, three Witness instances, a static metadata publisher, and a demo client into a single topology, with synthetic fixtures and a narrated walkthrough. Atlas is run separately and isn't in the Compose topology at v0.1.

Lab is not production deployment guidance, and the demo isn't a real integration with OpenCRVS, OpenSPP, DHIS2, OpenIMIS, MOSIP, or any other production system.
