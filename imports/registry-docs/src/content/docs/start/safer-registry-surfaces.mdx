Registry Stack is infrastructure for safer registry interoperability.
It helps human teams, exchange layers, workflow engines, and AI-assisted tools work against a
clear registry surface instead of files, undocumented tables, broad APIs, or copied extracts.

The goal is not to make registries available to AI.
The goal is to make registry use more explicit, governed, and reviewable for every caller.
AI and automation make that need more urgent because they make integration faster.

## The existing problem

Registry-like data often exists before the service surface around it exists.
An institution may have the right spreadsheet, extract, database, or legacy application, but not
the metadata, API contract, access policy, disclosure rule, or audit trail that another service
needs.

When that happens, policy stays in documents and meetings.
The runtime system does not know which caller is allowed to use which route, which fields can
leave the source boundary, which evidence response is acceptable, or which audit event must be
created.

That is the gap Registry Stack targets: make registry use operational, not only documented.

## What automation changes

Automation makes integrations easier to create.
AI-assisted tools can help teams read API documentation, draft client code, compare schemas,
generate tests, and explain data models.
Workflow engines can move data between systems with less custom code.

That helps good interoperability.
It also means weak contracts, broad routes, and wrong semantic assumptions can spread faster.
A team may be able to build an integration faster than it can understand the data, the authority
for access, the disclosure risk, and the meaning of each field.

The answer is not to avoid automation.
The answer is to give both people and automated tools a registry surface that is narrow, explicit,
and enforceable by default.

## What Registry Stack enables

Registry Stack provides registry-facing tools that make existing data usable and safer to depend on.

- Registry metadata gives humans and tools structured information about datasets, entities, fields,
  schemas, policies, services, and evidence offerings.
- Registry Relay exposes configured read-only routes over existing sources.
  A caller cannot widen the route at request time to fetch extra fields or broader source records.
- Registry Witness can answer configured evidence questions instead of forcing source-record access.
- Audit behavior records trust-critical access and evidence events for review.
- Semantic metadata makes meaning, sensitivity, and assumptions visible before integration code is
  written.

These are useful for ordinary integrations today.
They also give AI-assisted work and workflow automation safer material to operate on.

## From policy text to runtime checks

Safeguards are weak when they exist only as principles, policies, or implementation notes.
Registry Stack helps enforce policy at the point of access.

A registry surface can require a caller scope before a route is used.
It can expose only declared fields.
It can return a configured evidence response instead of a full record.
It can make purpose, policy, and disclosure rules inspectable before integration work starts.
It can record the access or evidence event for later review.

That does not remove the need for legal authority, governance, human review, or appeal paths.
It makes those decisions easier to operationalize because the registry surface can carry and
enforce part of the policy.

## The question shifts

Without these controls, the main question is whether two systems can connect.
For modern interoperability, that question is too weak.

The better question is whether this request is legitimate, scoped, meaningful, minimal, and
auditable.
That requires a registry surface that can answer practical review questions.

- Which entity or evidence response is being requested?
- Which purpose, policy, and scope apply?
- Which fields or claims can leave the source boundary?
- Which semantic definition is being used?
- Which audit event records the access or evidence decision?

## What remains outside Registry Stack

Registry Stack does not provide chat interfaces, form-filling assistants, model governance,
generated code review, or automated eligibility decisions.
It also does not replace human escalation, appeal, correction, accessibility testing, or the
governance needed around high-impact public services.

If AI is part of a service design, the first question remains whether AI helps the public-service
task.
Responsible non-adoption is a valid outcome.

AI remains one source of urgency for these docs, not a product pillar.
The product focus remains Registry Relay, Registry Witness, and registry metadata.

## How to read the rest

Use this page as a reason to inspect the registry controls, not as a separate AI product track.
For existing sources, start with Registry Relay and metadata.
For over-sharing risk, start with Registry Witness and evidence issuance.
For implementation review, inspect policy, scope, semantic metadata, and audit behavior before
reviewing integration code or workflow automation.
