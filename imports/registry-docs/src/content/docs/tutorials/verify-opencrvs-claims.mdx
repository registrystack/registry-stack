Use this tutorial when you have OpenCRVS client credentials and want the shortest local
Registry Notary path to a live claim check.
`registryctl` generates the Notary project, keeps your OpenCRVS credentials in `.env`, and writes
local Notary demo credentials under `secrets/`.
The generated project evaluates the starter `opencrvs-birth-record-exists` claim.

This tutorial does not configure governed policy, production signing, or wallet issuance.
The generated project includes one local demo SD-JWT VC profile for the starter claim.
OpenCRVS remains the evidence source, and Registry Notary signs the demo credential.

Estimated time: about 10 minutes after you have credentials and a test UIN.

## Before you start

You need:

- Docker with Compose v2.20 or later.
- `registryctl` with `init notary --recipe opencrvs-dci` support.
- `curl` and `jq` for the credential issuance check.
- OpenCRVS OAuth client credentials for a test environment.
- A known test UIN for the live smoke test.

Install `registryctl` if you have not already:

```sh
curl -fsSL https://raw.githubusercontent.com/jeremi/registry-registryctl/v0.2.0/install.sh | REGISTRYCTL_VERSION=v0.2.0 sh
registryctl --version
```

## Prepare the credential file

Create a working directory with one `.env` file:

```sh
mkdir opencrvs-notary
cd opencrvs-notary
```

Add your source credentials to `.env`. The generated Notary config uses these
connector-level env names:

```dotenv
DCI_BASE_URL=https://opencrvs.example.gov
DCI_CLIENT_ID=<opencrvs-client-id>
DCI_CLIENT_SECRET=<opencrvs-client-secret>
```

The recipe also accepts these OpenCRVS-specific aliases from existing demo
credential files:

```dotenv
OPENCRVS_DCI_BASE_URL=https://opencrvs.example.gov
OPENCRVS_DCI_CLIENT_ID=<opencrvs-client-id>
OPENCRVS_DCI_CLIENT_SECRET=<opencrvs-client-secret>
```

Keep `.env` private.
Do not paste bearer tokens into the file for this recipe.
Registry Notary uses the OAuth client-credentials settings to fetch a source token.

## Generate the Notary project

Run the recipe in the directory that contains `.env`:

```sh
registryctl init notary . --recipe opencrvs-dci --env-file .env
```

The generated project contains:

```text
opencrvs-notary/
  .env
  .gitignore
  registryctl.yaml
  compose.yaml
  README.md
  notary/
    config.yaml
  secrets/
    notary.local.env
  output/
  bruno/
```

Your `.env` file is preserved.
`secrets/notary.local.env` contains local Notary API, audit, replay, and demo issuer values.
The generated `compose.yaml` loads both files.

## Validate and start

Run the local doctor before starting services:

```sh
registryctl doctor --profile local --format json
```

`doctor` checks the generated project and delegates Registry Notary config validation when a
compatible `registry-notary` binary is available.
The report must not print your OpenCRVS client secret.

Start the generated stack:

```sh
registryctl start
```

Registry Notary listens on:

```text
http://127.0.0.1:4255
```

Open the local API docs when you want to inspect the generated routes:

```sh
registryctl notary open
```

## Run the first live claim

Set the known test UIN in your shell:

```sh
export OPENCRVS_DEMO_SUBJECT_UIN='<test-uin>'
```

Run the live smoke test:

```sh
registryctl notary smoke --target-env OPENCRVS_DEMO_SUBJECT_UIN
```

You can also pass the test UIN directly:

```sh
registryctl notary smoke --target-id <test-uin>
```

The smoke test checks health, readiness, authenticated Notary access, and the live OpenCRVS-backed
evaluation.
`registryctl` writes the detailed report to:

```text
output/notary-smoke-results.json
```

If you run the smoke test without `--target-id` or `--target-env`, the generated project can still
pass structural checks, but the live OpenCRVS evaluation is not run.

## Read the result safely

Inspect the redacted smoke report:

```sh
jq '.checks[] | {name, status}' output/notary-smoke-results.json
```

Do not paste the full report into support channels unless you have reviewed it.
The smoke report is designed to avoid raw OpenCRVS rows, target UINs, and secret values, but local
operators remain responsible for handling evidence reports as sensitive artifacts.

## Issue the demo credential

The smoke command proves the live claim works, but it does not print the raw evaluation response.
Run one explicit evaluation to capture the stored evaluation id:

```sh
NOTARY_API_KEY="$(
  awk -F= '$1 == "NOTARY_API_KEY_RAW" { sub(/^[^=]*=/, ""); print; exit }' \
    secrets/notary.local.env
)"

EVALUATION_ID="$(
  curl -fsS http://127.0.0.1:4255/v1/evaluations \
    -H "content-type: application/json" \
    -H "accept: application/vnd.registry-notary.claim-result+json" \
    -H "x-api-key: $NOTARY_API_KEY" \
    -H "data-purpose: https://example.local/purpose/tutorial" \
    -d '{
      "target": {
        "type": "person",
        "identifiers": [
          {
            "scheme": "UIN",
            "value": "'"$OPENCRVS_DEMO_SUBJECT_UIN"'"
          }
        ]
      },
      "claims": ["opencrvs-birth-record-exists"],
      "disclosure": "predicate",
      "format": "application/dc+sd-jwt",
      "purpose": "https://example.local/purpose/tutorial"
    }' | jq -r '.results[0].evaluation_id'
)"

test -n "$EVALUATION_ID"
```

Issue the local demo SD-JWT VC from that evaluation:

```sh
curl -fsS http://127.0.0.1:4255/v1/credentials \
  -H "content-type: application/json" \
  -H "x-api-key: $NOTARY_API_KEY" \
  -H "data-purpose: https://example.local/purpose/tutorial" \
  -d '{
    "evaluation_id": "'"$EVALUATION_ID"'",
    "credential_profile": "opencrvs_birth_record_sd_jwt",
    "claims": ["opencrvs-birth-record-exists"],
    "disclosure": "predicate",
    "format": "application/dc+sd-jwt"
  }' | jq '{
    credential_profile,
    format,
    credential_present: (.credential != null)
  }'
```

Expected result:

```json
{
  "credential_profile": "opencrvs_birth_record_sd_jwt",
  "format": "application/dc+sd-jwt",
  "credential_present": true
}
```

Do not paste the full credential into support channels.
It is a local demo credential signed by Registry Notary, not an OpenCRVS-issued credential.

## What the generated config does

The generated `notary/config.yaml` configures:

- An OpenCRVS source connection named `opencrvs_crvs`.
- OAuth client-credentials auth using `DCI_CLIENT_ID` and `DCI_CLIENT_SECRET`.
- The `/registry/sync/search` source path used by this OpenCRVS connection.
- A UIN lookup for birth registrations.
- The `opencrvs-birth-record-exists` claim.
- `predicate` disclosure as the default response.
- The `opencrvs_birth_record_sd_jwt` local demo credential profile.
- Local Notary signing material for demo issuance, not production trust.

For the mental model behind these blocks, see the Registry Notary
[OpenCRVS onboarding model](https://docs.registrystack.org/products/registry-notary/opencrvs-onboarding/).
