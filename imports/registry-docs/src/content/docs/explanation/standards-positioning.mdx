The registry family is not a new standard: it wraps practical implementation surfaces around existing ones and ties each claim to an owning project and verifiable evidence.

## Per-standard stance

**DCAT and BRegDCAT-AP** (`emits`):
Registry Relay emits DCAT-shaped metadata at `/metadata/dcat` and a BRegDCAT-AP profile at `/metadata/dcat/bregdcat-ap`. Registry Manifest renders the same artifacts offline. Both are artifact claims: the published bundle uses DCAT and BRegDCAT-AP vocabulary terms. Profile-specific validation evidence has not yet been pinned.

**OGC API Records** (`emits`):
Registry Relay exposes OGC Records routes behind a feature flag. Registry Manifest renders OGC Records item bodies and includes conformance class declarations. The claim level is `emits` because a feature flag gates the routes and full conformance test results are not yet available.

**OGC API Features** (`aligns_with`):
A feature flag for GeoJSON support exists in Registry Relay. The claim is `aligns_with` because route evidence has not been pinned to a reviewed fixture.

**OpenAPI** (`emits`):
Registry Relay generates an OpenAPI document at `/openapi.json`. Registry Witness generates and exposes its OpenAPI document at `/openapi.json`. Registry Legend publishes pinned artifacts through Redoc rather than hand-writing endpoint reference pages.

**SHACL and JSON Schema** (`emits`):
Registry Relay exposes SHACL node shapes at `/metadata/shacl` and entity JSON Schemas at `/datasets/{dataset_id}/{entity}/schema`. Registry Manifest renders the same files into its static publication bundle. External SHACL validation via pySHACL or the SEMIC validator is available but not wired into CI.

**JSON-LD** (`emits`):
DCAT, BRegDCAT-AP, SHACL, ODRL, and CCCEV outputs from Registry Relay, Registry Manifest, and Registry Witness all use JSON-LD structure (`@context`, `@type`, `@id`). No broad RDF dataset conformance claim is made.

**SD-JWT VC** (`emits`):
Registry Witness owns SD-JWT VC issuance. The issuer signs with EdDSA and uses the `dc+sd-jwt` token type. Individual claim fields are wrapped in SHA-256-hashed disclosures. Credential claims stay on Witness pages unless Registry Relay is publishing provenance contexts or discovery metadata that points to Witness.

**W3C Verifiable Credentials Data Model** (`aligns_with`):
Registry Relay has VC-oriented provenance contexts and test fixtures. Registry Witness issues SD-JWT credentials. Neither has a pinned W3C VC Data Model profile or a validation suite result. The claim level stays at `aligns_with` until a reviewed profile lands.

**W3C Decentralized Identifiers** (`implements`):
Registry Relay publishes a `did:web` document at `/.well-known/did.json` when provenance is enabled. Registry Witness parses `did:jwk` values to resolve credential subjects. Both satisfy normative requirements from DID Core 1.0. The claim covers `did:web` and `did:jwk` only; no other DID methods are supported.

**CCCEV** (`emits`):
Registry Witness renders CCCEV-shaped JSON-LD for claim evaluation results, emitting `cccev:Evidence` nodes with `cccev:isProvidedBy`, `cccev:supportsRequirement`, and `cccev:supportsValue` properties. The media type is `application/ld+json; profile="cccev"`. Profile conformance is not claimed.

**ODRL** (`emits`):
Registry Manifest renders ODRL policy documents with `odrl:Policy`, `odrl:Rule`, and `odrl:Constraint` terms. Registry Relay exposes dataset-scoped ODRL Offers at `/metadata/datasets/{dataset_id}/policy`. Relay publishes these policies as metadata for downstream governance; it does not evaluate or enforce them at runtime.

**PROV-O** (`inspired_by`):
Provenance concepts appear in Relay and Witness internals, but neither emits standalone PROV-O vocabulary terms. The claim stays at `inspired_by` until PROV-O terms are emitted or a reviewed mapping is available.

**GovStack Digital Registries** (`compares_against`):
Registry Relay is not positioned as a GovStack conformant system. The GovStack Digital Registries Building Block describes a single uniform CRUD platform; Registry Relay explores a protected consultation-only gateway model instead.

**SP DCI** (`maps_to`):
Registry Relay exposes an SP DCI sync adapter behind a feature flag. Registry Witness ships an HTTP source connector that maps SP DCI search and sync responses into claim evaluation inputs. Claims stay narrow until reviewed official SP DCI fixtures are available.
