The five projects connect through portable metadata, runtime HTTP APIs, and published discovery artifacts. Each owns a bounded responsibility and does not depend on the internals of another.

The five projects:

- **[Registry Manifest](../../projects/registry-manifest/)**: portable metadata compiler and renderer.
- **[Registry Relay](../../projects/registry-relay/)**: protected, read-only consultation gateway.
- **[Registry Witness](../../projects/registry-witness/)**: claim evaluation, attestation, and credential issuance.
- **[Registry Atlas](../../projects/registry-atlas/)**: inspection workbench for published discovery artifacts.
- **[Registry Lab](../../projects/registry-lab/)**: compose-based runnable demo of the full topology.

## Data and contract flow

1. A metadata manifest (`registry-manifest/v1` schema) describes datasets, entities, fields, policies, and evidence offerings.
2. Registry Manifest compiles and validates the manifest, then renders a static discovery bundle (catalog, DCAT, SHACL, JSON Schemas, OGC Records items, policies, evidence-offering metadata, and an `index.json`).
3. Registry Relay starts with runtime config that binds the manifest's logical datasets and entities to actual data sources. Clients reach entity routes, metadata routes, and evidence-offering endpoints through configured auth.
4. Registry Witness evaluates claims against configured HTTP sources (Registry Data API or SP DCI connectors). It renders results as `application/vnd.registry-witness.claim-result+json`, CCCEV-shaped JSON-LD (`application/ld+json; profile="cccev"`), or SD-JWT VCs (`application/dc+sd-jwt`).
5. Registry Atlas fetches published catalog and discovery artifacts through its proxy, classifies them by artifact kind, and runs strict capability queries over the indexed metadata terms.

For what each project does and does not own, see [ownership boundaries](../../map/ownership-boundaries/).
