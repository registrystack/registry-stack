The registry family supports DPI safeguards by making registry consultation, metadata publication, claim evaluation, and discovery review constrained, explicit, auditable, and visible through standards-shaped artifacts.

## Reference framework

The [Universal DPI Safeguards website](https://www.dpi-safeguards.org/) is the online reference for the safeguards initiative and framework.
It presents the framework as a multi-stakeholder effort to help DPI implementations mitigate risk, advance safe and inclusive services, and build trust across countries.

This page compares the registry family to the framework at the technical implementation layer.
The registry family is not a complete safeguards program, and alignment with this page is not certification.

## Technical alignment

The registry family is most relevant where safeguards depend on controlled access, data minimization, purpose-bound exchange, auditability, reviewable metadata, and verifiable provenance.
Those are technical foundations that can support a DPI program's governance work.

| Safeguards concern | Registry family support |
| --- | --- |
| Privacy by design | Registry Relay can expose only configured fields, require filters, separate metadata from runtime bindings, and verify evidence without returning raw registry rows by default. |
| Data security by design | Registry Relay supports API key or OIDC authentication, independent scopes, private admin routes, read-only source bindings, source-required scopes, body limits, and timeouts. |
| Transparency and accountability | Registry Manifest and Registry Relay can publish OpenAPI, metadata routes, schemas, evidence offering descriptions, audit records, stable error codes, and provenance artifacts. |
| Open assets and interoperability | Registry Manifest, Registry Relay, and Registry Witness produce or reference standards-shaped artifacts such as DCAT, SHACL, JSON Schema, ODRL, OpenAPI, and SD-JWT VC profiles. |
| Evidence-based improvement | Audit logs, aggregate disclosure records, validation reports, provenance receipts, and standards registers can provide material for review and impact assessment. |

This alignment is strongest at the implementation-evidence layer.
It helps reviewers see what a registry system exposes, which project owns each surface, and which controls are available before deployment decisions are made.

## Project roles

Registry Manifest provides the portable metadata layer.
It can describe datasets, entities, fields, codelists, policy metadata, schemas, and evidence offerings without exposing source paths, table ids, backend URLs, or secrets.

Registry Relay provides the protected consultation layer.
It turns configured registry sources into read-only APIs with scoped access, declared filters, purpose context, audit records, aggregate disclosure controls, and metadata filtered by caller visibility.

Registry Witness provides the claim evaluation layer.
It evaluates configured claims against registry sources, applies disclosure policy, records redacted audit events, and can issue credentials where a reviewed trust model allows it.

Registry Atlas provides the inspection layer.
It helps implementers and reviewers inspect published discovery artifacts and capability descriptions, making registry claims easier to verify and discuss.

## Governance boundary

The Universal DPI Safeguards Framework covers more than software controls.
A DPI program still needs lawful basis, public governance, procurement discipline, data protection operations, accessibility, remedies, oversight, security audits, and community participation.

The registry family does not obtain consent, approve data-sharing agreements, run grievance channels, certify production sources, or make final eligibility decisions.
It also does not replace a data protection impact assessment, independent security review, or public consultation process.

The intended boundary is clear: registry components produce evidence that responsible institutions can review.
They do not hide governance decisions inside software.

## Practical use

A DPI team can use the registry family when it needs to expose sensitive registry facts to authorized systems without making raw registry data broadly available.
Good-fit use cases include cross-ministry intake checks, social protection eligibility evidence, civil registration fact verification, interagency aggregate statistics, and signed evidence receipts for downstream review.

Poor-fit use cases include unrestricted public data portals, generic data warehouses, ad hoc SQL access, unrestricted record export, and service decision engines without external governance.

Registry Legend presents the registry family as a technical safeguards component.
The broader safeguards outcome depends on the institution that governs, deploys, audits, and operates the DPI system.
