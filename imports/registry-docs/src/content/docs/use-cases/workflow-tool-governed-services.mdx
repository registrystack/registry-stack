## Starting situation

A farmer registry holds 80,000 smallholder households across four regions in a PostgreSQL table.
An extension-service workflow tool issues input vouchers to registered farmers, scoped by region
and active-season programme.
The workflow tool needs to confirm registration status and region assignment for each applicant.
The registry operator will not give the tool a database credential; it must call a governed
service and store only the result its process is authorized to keep.

## What you set up

Deploy Registry Relay with a consultation route that accepts a farmer registration number and
returns the authorized fields: registration status, assigned region, and season flag.
Scope the route so the extension-service caller can read only those fields and only for the
regions its operator role covers.

Configure a Registry Witness claim for the active-registration predicate if the workflow needs a
signed yes/no answer rather than a field set.
Use Registry Manifest to publish service, schema, and policy metadata that the workflow team
can read before starting integration.

The workflow tool remains a consumer.
It sends an authorized request, receives the allowed response, records the workflow decision or
receipt, and leaves source records in the farmer registry.

## What a caller does

The workflow tool sends an authorized GET or POST to the Relay consultation route, passing the
farmer registration number and the caller's scoped token.
If the workflow needs a signed fact, it calls the Witness evaluation endpoint instead and
receives a predicate or credential.

A multi-step flow is possible: use Manifest metadata to discover the service contract, use
Relay to retrieve the region field, use Witness to receive a signed active-registration
predicate, and record both results in the workflow audit log.

## What the caller gets back

From Relay: a JSON response containing the authorized fields and a provenance object naming the
issuing registry and evaluation timestamp.

From Witness: a signed evaluation containing the subject identifier, the predicate result, the
evaluation timestamp, and the issuer identifier.

The full household record, internal registry notes, and correction history do not appear in
either response.

## What is in scope, what is not

This pattern covers: a workflow tool calling Relay for a scoped field response, calling Witness
for a signed fact, and using Manifest metadata to discover the service contract.

It does not cover: the workflow tool acting as a registry, storing source records, or inheriting
registry-of-record status.
Caller identity, purpose limitation, data retention, audit review, and rules for what the
workflow may store are deployment-specific governance controls outside the registry family.
