## Starting situation

A professional licensing authority records active licenses for 35,000 practitioners across six
regulated sectors in a set of spreadsheets and a legacy licensing application.
Leadership has committed to moving to a modern platform in three years, but no migration contract
is signed yet.
The authority wants to use the preparation window to describe the data estate, expose schema
inconsistencies, and produce artifacts that will be useful regardless of which platform it
eventually adopts.

## What you set up

Write Registry Manifest profile fixtures that describe each sector's license dataset: the
publisher, the dataset, the data service (even if it is currently only an export), the field
schema, and the access policy.

Run Manifest to render static metadata bundles: a DCAT catalog, JSON Schema per sector, SHACL
shapes that make field expectations explicit, and ODRL policy stubs.

Run Registry Atlas against the published bundles.
Atlas reports which artifacts are present, which are missing, and which fields are ambiguous
across sectors.
The inspection report becomes the baseline for migration planning.

Registry Platform contains shared primitives that other registry family components use, but a
complete future registry platform is a separate undertaking.
The preparation work described here uses only Manifest and Atlas.

## What a caller does

In this use case the "caller" is the authority's own team.
They iterate: run Manifest to render, run Atlas to inspect, review the report, correct profile
fixtures, and re-render until the metadata accurately describes the estate.

External partners can be given the published metadata bundle to read before any production
integration is agreed.
That gives them a contract to comment on before the migration begins.

## What the caller gets back

A set of metadata artifacts: a catalog of existing registry surfaces and responsible
organizations, entity schemas and SHACL shapes that make record expectations explicit, service
and requirement metadata for discovery, policy stubs that state intended access constraints, and
Atlas inspection reports that show gaps, ambiguous fields, and missing documents.

These artifacts serve migration planning.
They also serve non-migration work: partner onboarding, API documentation, governance review,
and registry inspection can all use the same published metadata while the source system
remains in place.

## What is in scope, what is not

This pattern covers: producing metadata and quality evidence for an existing data estate using
Manifest and Atlas.

It does not claim that Registry Platform currently provides a complete production registry
system, a migration engine, or an automatic data-quality remediation service.
The future platform is a separate undertaking; the preparation work here is independent of
when or whether that platform is built.
