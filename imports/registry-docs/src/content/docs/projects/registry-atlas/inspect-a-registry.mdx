Use this page to point Atlas at a published registry, explore the catalog and discovery
artifacts it exposes, and check whether its published capabilities match what the strict
capability engine can verify.

## When to use this

Use these steps when you want to:

- Confirm that a registry's metadata endpoint returns well-formed Data Catalog Vocabulary (DCAT)
  or BRegDCAT-AP documents before you write integration code.
- Browse the schemas, SHACL shapes, ODRL policies, and evidence offerings the registry publishes.
- Run a strict capability query to check whether the registry's metadata supports a specific
  information need (for example, farmer registration status or disability status).

## Prerequisites

- Atlas is running locally.
  Follow [Run Atlas locally](../run-locally/) first if you have not already.
- You have the metadata endpoint URL for the registry you want to inspect.
  For Registry Relay, this is the URL at the path `/metadata` (for example,
  `http://127.0.0.1:4242/metadata` for the local all-standards demo).
- Optional: a bearer token if the endpoint requires authorization.
  Bearer tokens are session-only in Atlas and are never persisted.

## Steps

### 1. Open Atlas

Open `http://127.0.0.1:5177` in a browser.

### 2. Enter the registry URL

Type the catalog endpoint URL into the URL field in the top bar.

For the Registry Relay all-standards demo, the URL is:

```text
http://127.0.0.1:4242/metadata
```

If the endpoint requires authorization, paste the bearer token into the session token field
next to the URL.
The token is forwarded to the proxy and is discarded when the browser tab closes.

### 3. Select a profile (optional)

Atlas infers the catalog profile from the `conformsTo` declaration in the catalog document
when one is present.
Supported profiles are DCAT-AP 3.0.0, DCAT-AP 2.1.1, BRegDCAT-AP, and the Registry Relay
publisher-specific profile.
If Atlas cannot infer the profile, select one manually from the profile dropdown.

### 4. Run discovery

Press **Discover**.
The button label comes from
[`src/App.tsx`](https://github.com/jeremi/registry-atlas/blob/main/src/App.tsx#L478),
where the submit button renders the text `Discover`.

The proxy fetches the catalog document and follows links to retrieve every artifact it finds:
SHACL shapes, JSON Schema files, ODRL policy documents, OpenAPI descriptions, and OGC API
Records landing pages.
Atlas classifies each fetched artifact into a semantic asset kind and assembles a
`DiscoveryReport`.

Discovery progress is visible in the loading state.
The proxy enforces an 8-second timeout per request and a 2 MB response size limit.
Large or slow registries may produce partial reports if artifacts time out.

### 5. Browse the Overview tab

The **Overview** tab shows:

- Total counts of discovered artifact types and semantic assets.
- Recognized profile and conformance status.
- Gaps or missing items (for example, missing policy documents or absent SHACL shapes).
- Follow-up evidence links.

Gaps are flagged explicitly.
A missing item in the overview means Atlas could not find or parse that artifact kind,
not necessarily that the registry does not publish it.
The artifact may be behind an auth-required endpoint, may use a content type the proxy
does not accept, or may be linked incorrectly.

### 6. Browse discovered assets in the Registry tab

Open the **Registry** tab for a searchable list of discovered datasets, data services,
distributions, and catalogs.

Select an asset to open its detail pane.
The detail pane shows:

- The asset kind (for example, `dataset`, `data_service`, `distribution`).
- Recognized metadata fields (title, description, publisher, conformsTo, license).
- Access methods: endpoint URLs, supported media types, authorization hints.
- Links to discovered schemas and SHACL shapes associated with this asset.

### 7. View schemas, policies, and evidence offerings

From the Registry tab or the Evidence tab, select a `json_schema`, `shacl`, or `policy` asset
to see the raw parsed document.

For SHACL shapes, Atlas shows the properties and constraints defined in the shape.
For JSON Schema files, Atlas shows the draft version and the declared properties.
For ODRL policy documents, Atlas shows the offer terms and constraints.

### 8. Run a strict capability query

Open the **Capabilities** tab.

The capability query panel accepts one or more **information needs**.
Each need requires at least one accepted term: a label, an IRI, or a field name that the strict
matcher is allowed to compare against discovered metadata.

The engine matches terms only after documented canonicalization:
whitespace trimming, Unicode normalization, and case folding for labels.
It does not expand synonyms, apply fuzzy matching, or consult a language model.
Use exact terms from the registry's published metadata, not natural-language descriptions.

For example, to check whether the registry publishes farmer registration status:

- Add a need named `farmer_status`.
- Add a required term with label `Farmer` (or the IRI published in the registry's DCAT catalog
  or SHACL shapes).

The engine produces candidate routes for each matched need.
Each candidate route includes:

- A confidence indicator.
- The access endpoint URL.
- The access kind (for example, `data_service`, `distribution`).
- Evidence references (the discovered artifact and field that produced the match).
- Gaps (for example, `LegalBasisUnknown`, `AuthorityUnknown`, `FreshnessUnknown`).
- Review flags that indicate where human or governance review is recommended.

A candidate route is not an authorization grant and is not a guarantee of access.
It is a discovery result backed by machine-verifiable evidence from the registry's published
metadata.
The full contract for how candidate routes are produced is in
[`SYSTEM_CAPABILITY_DISCOVERY_SPEC.md`](https://github.com/jeremi/registry-atlas/blob/main/SYSTEM_CAPABILITY_DISCOVERY_SPEC.md).

### 9. Inspect raw evidence

Open the **Evidence** tab to see the full `DiscoveryReport` JSON and the artifact tree.

The comparison view lets you separate:

- `core` view: only fields that Atlas recognized via standard DCAT, DCAT-AP, or BRegDCAT-AP vocabulary.
- `publisher` view: registry-specific metadata that Atlas found but did not map to a standard term.
- `diff` view: side-by-side comparison of the two.

The raw report is the portable evidence contract.
You can copy it, save it as a file, and pass it to the `semantic-asset-discovery-cli` for
offline analysis or to the capability query CLI.

## Verification

After completing the steps, confirm:

- The Overview tab shows at least one catalog asset and no parse errors.
- The Registry tab lists datasets or data services.
- The Capabilities tab returns candidate routes for at least one need when the registry
  publishes metadata with the terms you queried.
- The Evidence tab shows the `DiscoveryReport` JSON with a non-zero `asset_count`.

For the Registry Relay all-standards demo, the expected summary is 168 assets across 29 artifacts,
broken down as follows: 1 `api_description`, 3 `catalog`, 22 `class`, 44 `data_service`,
27 `dataset`, 44 `distribution`, and 27 `policy`.
These counts are verified against the bundled fixture at
[`fixtures/system-capability/registry-relay-all-standards.report.json`](https://github.com/jeremi/registry-atlas/blob/main/fixtures/system-capability/registry-relay-all-standards.report.json)
and match the expected values documented in
[`HOW_TO_TEST_V0_1.md`](https://github.com/jeremi/registry-atlas/blob/main/HOW_TO_TEST_V0_1.md#L128-L149).

## Troubleshooting

### Atlas shows `fetch-error` or `auth-required`

Check that the registry endpoint is reachable and that the URL is correct.
If the endpoint requires a bearer token, confirm the token is entered in the session token field.
If you are inspecting a local Registry Relay instance, confirm `ATLAS_PROXY_ALLOW_LOCAL=1` is
set or that `NODE_ENV` is not `production`.

### Discovery returns zero assets

The catalog document may use a format or content type the proxy does not accept.
The proxy allows `application/ld+json`, `application/json`, `application/schema+json`,
`application/vnd.oai.openapi+json`, `application/yaml`, and `text/*` responses.
RDF/XML and N-Quads are not yet supported.

### Capability query returns no matches

The strict engine requires exact term matches after canonicalization.
Open the **Evidence** tab and browse the `DiscoveryReport` assets to find the exact labels,
IRIs, or field names the registry publishes.
Use those values as your query terms.

### Response size limit exceeded

The proxy caps responses at 2 MB.
A registry that serves a very large SHACL graph or JSON Schema in a single document may
exceed this limit.
Set `ATLAS_PROXY_MAX_BYTES` to a higher value in the proxy environment if needed.
