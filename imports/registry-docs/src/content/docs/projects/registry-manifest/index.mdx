Write one YAML file, run a command, and get service catalogues, catalog metadata, validation
shapes, JSON Schema, forms, and policy documents out.
No server required.

Registry Manifest is a portable Rust workspace that lets you describe a registry once in YAML
and derive every standards-facing metadata artifact other systems need to find, understand, and
connect to it.
It does not serve HTTP, does not talk to databases, and does not handle credentials.

## When you'd use this

You have a registry or public-service workflow and need to publish machine-readable metadata:
a Core Public Service Vocabulary Application Profile (CPSV-AP) service catalogue, a Data Catalog
Vocabulary (DCAT) catalog, Shapes Constraint Language (SHACL) shapes, JSON Schema for API contract
documentation, a local form profile, or Open Digital Rights Language (ODRL) policies declaring
access terms.
You write one `metadata.yaml`, run `registry-manifest-cli publish`, and get a static directory
of all those artifacts ready to host anywhere.

If you need runtime metadata served over HTTP with per-caller scoping and authorization, that's
Registry Relay's job. Relay uses Registry Manifest's renderers internally; see
[map: ownership boundaries](../../map/ownership-boundaries/) for where the line falls.

## A minimal manifest

A manifest must declare `schema_version: registry-manifest/v1` at its root.
The following skeleton shows the required top-level keys:

```yaml
schema_version: registry-manifest/v1
catalog:
  id: my-registry
  base_url: https://registry.example.gov
  title:
    en: My Registry
  publisher:
    name: Example Authority
    iri: https://registry.example.gov/authority
    authority_type: eli:PublicAuthority
  participant_id: https://registry.example.gov/authority
datasets:
  - id: my-dataset
    title:
      en: My Dataset
    entities:
      - name: my_entity
        title:
          en: My Entity
        identifiers:
          - name: entity_id
            kind: local
        fields:
          - name: entity_id
            type: string
            required: true
requirements: []
evidence_types: []
public_services: []
forms: []
```

A manifest must not contain runtime bindings such as source file paths, table names, database
scopes, or backend credentials. Those live in Registry Relay configuration; the manifest is
portable and Relay-independent.

## The metadata-manifest model

The top-level struct is `MetadataManifest`, which contains:

- `CatalogManifest`: catalog title, publisher, base URL, application profiles, and standards versions in use.
- `DatasetManifest`: one entry per dataset, each carrying entities, policies, evidence offerings, codelists, and ODRL terms.
- `EntityManifest`: structural schema for a domain resource, including fields, identifiers, relationships, and vocabulary concept IRIs.
- `ServiceManifest`: CPSV-AP-facing public service metadata, channels, competent authority, requirements, produced resources, linked forms, and data services.
- `FormManifest`: local form-profile metadata for sections, fields, conditional visibility, repeating sections, validation references, and fulfillment modes.
- `EvidenceOfferingManifest`: description of a verification capability and the access path a client uses to reach it.
- `DatasetPolicyManifest` / `PolicyRuleManifest` / `PolicyDutyManifest` / `PolicyConstraintManifest`: dataset-scoped ODRL policy terms.
- `RequirementManifest` / `EvidenceTypeManifest`: Core Criterion and Core Evidence Vocabulary (CCCEV)-aligned requirement, evidence type, and grouped evidence type list definitions.
- `CodelistManifest`: enumerated value schemes with concept URIs.

Grouped evidence is explicit.
An `evidence_type_lists` entry under a requirement is one option group; all evidence types in that
list are required together.
Multiple lists on the same requirement are alternatives.

## Validation

When you run `validate` or `publish`, Registry Manifest performs multi-pass validation before
any rendering occurs:

1. Confirms `schema_version == "registry-manifest/v1"`.
2. Validates catalog fields: HTTP base URL, title, publisher, and supported application profiles list.
3. Validates requirement and evidence-type reference integrity.
4. Validates grouped evidence type lists, including duplicate list IDs, empty lists, unknown
   evidence types, and evidence types that do not prove the owning requirement.
5. Validates public service, channel, authority, form, and data-service references.
6. Validates per-dataset entities, fields, relationships, identifiers, codelists, cardinality strings, and ODRL policy terms.
7. Validates evidence-offering references.

All errors are collected and returned together so a single run surfaces every problem in the
file. You do not need to fix one error at a time and re-run.

Validation also expands vocabulary prefixes (for example, `person:Person.identifier` becomes a
full IRI) and resolves codelist concept references before any renderer runs.

## Renderers

Registry Manifest ships pure renderer functions. Each renderer takes a compiled manifest and
returns a serialized document. No renderer makes network calls or reads files beyond the manifest
input.

| Output format | `--format` value |
| --- | --- |
| Catalog JSON | `catalog` |
| DCAT JSON-LD | `dcat` |
| CPSV-AP service catalogue JSON-LD | `cpsv-ap` |
| BRegDCAT-AP 2.00 JSON-LD | `bregdcat-ap` |
| SHACL node shapes JSON-LD | `shacl` |
| JSON Schema Draft 2020-12 | `json-schema` |
| Form JSON Schema Draft 2020-12 | `form-json-schema` |
| OGC API Records item bodies JSON | `ogc-records` |
| ODRL policy collection JSON-LD | `policies` |
| Per-dataset ODRL policy document JSON-LD | `policy` |
| Evidence offerings collection JSON | `evidence-offerings` |
| Single evidence offering JSON | `evidence-offering` |

Not yet implemented: GovStack DR BB, SP DCI, and standalone PROV-O.

The `dcat` format dispatches based on an optional `--profile` flag: `--profile dcat` or
`--profile dcat-ap` renders base DCAT; `--profile bregdcat-ap` renders the BRegDCAT-AP variant.
Omitting `--profile` with `--format dcat` defaults to base DCAT.

## The CLI surface

`registry-manifest-cli` exposes four subcommands:

- `validate`: parse and validate a manifest file. Exits non-zero on any validation error.
- `render`: compile a manifest and write one renderer's output to stdout. Accepts `--format`,
  `--profile`, `--dataset`, `--entity`, and `--offering` selectors.
- `publish`: compile a manifest, run all renderers, write every artifact to an output directory,
  and generate an `index.json` manifest of the bundle.
- `validate-profiles`: scan a `profiles/` directory for profile descriptors, validate their
  schema, and validate all referenced fixture manifests.

The [validate-and-render how-to](./validate-and-render/) covers the `validate`, `render`, and
`publish` subcommands in steps.
The [profile fixtures how-to](./profile-fixtures/) covers `validate-profiles`.

## Profile fixtures

The `profiles/` directory contains non-normative profile descriptors and fixture manifests.
A profile descriptor (schema version `registry-manifest-profile/v1`) declares:

- Required and optional concept IRIs a conforming manifest must reference.
- Required identifiers and codelists with expected code values.
- Cardinality expectations per entity field.
- Runtime-only keys that must not appear in a portable fixture manifest.

Four example profiles ship in the repository: `example-civil-registration`,
`example-social-benefits`, `example-person-schema`, and `example-benefits-sync`.
These are examples until reviewed against official OpenCRVS, OpenSPP, OpenIMIS, SP DCI, or
maintainer-provided artifacts.

Four additional subdirectories exist in `profiles/` for
[`opencrvs`](https://github.com/jeremi/registry-manifest/tree/main/profiles/opencrvs/),
[`openimis`](https://github.com/jeremi/registry-manifest/tree/main/profiles/openimis/),
[`openspp`](https://github.com/jeremi/registry-manifest/tree/main/profiles/openspp/), and
[`spdci`](https://github.com/jeremi/registry-manifest/tree/main/profiles/spdci/).
Each contains only a `README.md` that marks it as a placeholder pending official review.
No `profile.yaml` or fixture files are present in any of these directories.

## Status and v0 caveats

Registry Manifest is at v0.1.
The manifest schema version `registry-manifest/v1` is enforced in code, but the schema itself
may still change between v0 releases.
Profile fixtures in `profiles/` are non-normative examples.
The static publication bundle format (including `index.json` schema version
`registry-manifest-index/v1`) is not yet a stable contract.

The test suite includes golden-fixture assertions for the older example profiles and a shared
CPSV-AP fixture contract used by Atlas service-first discovery tests.
