A registry API can be technically useful and still disclose more than a caller needs.
This page explains why safer registry exchange often requires purpose-appropriate
evidence instead of default full-record sharing.

## The problem

A service may need to know that a company is active, a license is valid, a facility is
registered, or a person satisfies a configured rule.
If the only available API returns the whole registry record, the caller receives data that
is not necessary for the immediate purpose.

Over-sharing can also happen inside apparently narrow APIs.
A route can expose enough fields, identifiers, relationships, or free-text values to let a
caller infer more than the service requires.

## Why it matters

Data minimization is easier to review when the service surface matches the question.
A predicate, status, selected disclosure credential, or attested extract can reduce what
crosses the boundary while still giving the caller useful evidence.

Over-sharing creates downstream obligations.
The caller must protect, retain, and explain data that the service may not have needed to
receive.
The registry owner must trust that every caller handles broad records correctly.

## Why common approaches stall

Role-based access to whole records can reduce the number of callers.
It does not answer whether a specific purpose needs the full record.

Custom minimized endpoints can help for one workflow.
They stall when each new evidence question requires another bespoke endpoint, another
review cycle, and another undocumented disclosure rule.

## What Registry Stack changes

Registry Stack separates read access from evidence exchange.
Registry Relay can expose protected entity routes with configured field projection and
scope checks.
Registry Witness can evaluate configured claims and return a claim-result JSON document,
CCCEV-shaped JSON-LD, or an SD-JWT VC credential where configured.
Registry Manifest can publish evidence-offering metadata so a caller can discover which
evidence path exists.

That model lets the deployment choose the evidence shape that fits the purpose instead of
making the full source record the default integration object.

## What is built today

Registry Relay documents declared field projection, caller scopes, and evidence-offering
metadata routes.
Registry Witness documents disclosure modes (`value`, `predicate`, and `redacted`), claim
evaluation, CCCEV-shaped output, direct credential issuance, and OID4VCI issuance.
Registry Platform provides shared primitives used by Relay and Witness for audit,
authentication, HTTP safety, and SD-JWT handling.

See [evidence issuance](../../explanation/evidence-issuance/) and
[Registry Witness reference](../../projects/registry-witness/reference/).

## What is future-facing or out of scope

Registry Stack does not decide whether a caller is legally entitled to a fact.
The deployment must define policy, configure allowed disclosure modes, and decide which
evidence shape is appropriate for the purpose.

Lineage evidence, chain-of-title evidence, and broader sector-specific evidence profiles
are future-facing unless a source repo documents and tests the pattern.
