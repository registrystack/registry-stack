# Decentralized Evidence Demo Spec

## Purpose

Build a Docker Compose demo that shows the post-split architecture as a small
decentralized network of registry authorities and evidence services.

The demo must make these boundaries visible:

- Registry Relay instances expose registry data and metadata.
- Evidence Server instances compute evidence independently from Relay.
- Clients discover evidence capabilities through metadata, then call the
  advertised Evidence Server endpoint directly.
- Registry data lives in mounted source extracts, not in demo database
  containers.
- Evidence Server reads registry data only over HTTP.

This is a simulated functional-domain demo. It must not use product names such
as OpenCRVS or OpenSPP until the demo is wired to real product instances or
reviewed product-specific adapters.

## Grounded Review

Registry Relay is a protected consultation gateway over configured sources. For
this demo, use file-backed sources:

- CSV, XLSX, and Parquet are supported by Relay, and v1 uses all three so the
  demo shows the real Relay source model.
- Source files are mounted read-only into Relay containers.
- Relay snapshots source rows into a writable internal Parquet cache.
- Relay exposes entity-shaped APIs and metadata from runtime config and
  portable metadata manifests.
- Row reads that require purpose must send `Data-Purpose`.

Evidence Server is a separate process:

- It owns claim definitions, rules, disclosure policy, rendering, auth, audit,
  and source connector config.
- It calls Relay over HTTP through the Registry Data API or DCI-compatible
  routes.
- It must not read source files directly.

DCI routes in Relay are feature-gated by `spdci-api-standards`. If any demo
Evidence Server config uses `connector: dci`, the demo Relay image must be
built with that feature. The existing production Relay Dockerfile builds
without feature flags, so the demo must use a demo-specific image build or
Dockerfile path.

## Non-Goals

- Do not add database containers for v1.
- Do not collapse the demo to a bilateral two-registry proof. V1 must include
  more than two registry authorities.
- Do not imply real OpenCRVS, OpenSPP, DHIS2, OpenIMIS, or other product
  support.
- Do not make Relay the central verification service.
- Do not allow unauthenticated evidence or row APIs.
- Do not move source data into Evidence Server.
- Do not put source paths, scopes, table ids, or backend runtime details into
  portable metadata manifests.
- Do not introduce Kubernetes, OIDC, external IdPs, or external cloud
  dependencies for v1.

## V1 Topology

Compose services:

```text
demo-client
  |
  | fetches metadata and service discovery
  v
civil-registry-relay
social-protection-registry-relay
health-registry-relay
static-metadata-publisher
  |
  | metadata discovery, HTTP source APIs, Registry Data API, or DCI-compatible routes
  v
civil-evidence-server
social-protection-evidence-server
shared-eligibility-evidence-server
```

This is intentionally more than a two-node proof. Three Relay authorities plus
multiple Evidence Servers make the decentralized boundary legible: registry
operators, metadata publishers, and evidence-service operators are separate
roles, and one shared Evidence Server can consume more than one authority.

Each Relay instance is an independent registry authority:

- its own runtime config;
- its own portable metadata manifest;
- its own mounted source extract;
- its own API key hash env vars;
- its own writable cache volume;
- stdout audit events.

Each Evidence Server instance is an independent verifier:

- its own config;
- its own auth credentials;
- its own source connection list;
- stdout audit events;
- one or more claims using source data over HTTP.

`static-metadata-publisher` is a separate static HTTP service that serves a
metadata bundle generated by `registry-manifest-cli`. It proves that metadata
publication can outlive or sit beside Relay runtime APIs.

## Runtime Services

### civil-registry-relay

Functional role:

- simulate a civil registry authority;
- expose civil-person or civil-status records;
- publish metadata for the civil registry dataset and evidence offerings.

Source format:

- CSV.

Example source rows:

- `national_id`;
- `given_name`;
- `surname`;
- `birth_date`;
- `civil_status`;
- `deceased`;
- `district`.

Route shape:

- use Relay's real entity route pattern:
  `/datasets/{dataset_id}/{entity}`;
- the demo config may choose `dataset_id: civil_registry` and
  `entity: civil_person`, but the spec does not require those names;
- row reads must include `Data-Purpose` when the entity config requires it.

Expected APIs:

- `/health`;
- `/ready`;
- `/datasets`;
- `/datasets/{dataset_id}`;
- `/datasets/{dataset_id}/{entity}`;
- `/metadata`;
- `/metadata/evidence-offerings`;
- optional `/dci/{registry}/registry/sync/search` only when the demo image is
  built with `spdci-api-standards`.

### social-protection-registry-relay

Functional role:

- simulate a social protection registry authority;
- expose households, persons, and program enrollments;
- publish metadata for social protection datasets and evidence offerings.

Source format:

- XLSX, ideally multi-sheet.

Example sheets:

- `Households`;
- `Persons`;
- `Enrollments`.

Example downstream claims:

- household exists;
- person is enrolled in program;
- program enrollment is active;
- eligibility band or poverty score is under a threshold.

Route shape:

- use Relay's real entity route pattern:
  `/datasets/{dataset_id}/{entity}`;
- row reads must include `Data-Purpose` when required by config.

### health-registry-relay

Functional role:

- simulate a health facility or health workforce registry authority;
- expose facility, practitioner, license, or service availability records;
- publish metadata for health registry datasets and evidence offerings.

Source format:

- Parquet preferred for v1.
- XLSX fallback is allowed only if Parquet fixture generation blocks
  implementation, and the README must record the exact blocker.

Example downstream claims:

- facility license is active;
- service is available in a district;
- practitioner credential is active.

Route shape:

- use Relay's real entity route pattern:
  `/datasets/{dataset_id}/{entity}`;
- row reads must include `Data-Purpose` when required by config.

### civil-evidence-server

Functional role:

- compute civil evidence by querying `civil-registry-relay`.

Example claims:

- `date-of-birth`;
- `civil-status`;
- `person-is-alive`.

Source connections:

- one connection to `http://civil-registry-relay:8080`;
- one source token env var specific to the civil Relay connection.

### social-protection-evidence-server

Functional role:

- compute social protection evidence by querying
  `social-protection-registry-relay`.

Example claims:

- `program-enrollment-status`;
- `household-eligibility-band`;
- `beneficiary-active`.

Source connections:

- one connection to `http://social-protection-registry-relay:8080`;
- one source token env var specific to the social protection Relay connection.

### shared-eligibility-evidence-server

Functional role:

- demonstrate that one verifier can consume multiple registry authorities.

Required claim shape:

- one CEL-derived claim;
- claim dependencies must read from at least two Relay authorities;
- each source connection must use a distinct `token_env`.

Source connections:

- one connection to `http://civil-registry-relay:8080`;
- one connection to `http://social-protection-registry-relay:8080`;
- one connection to `http://health-registry-relay:8080` if the shared health
  claim is implemented in v1;
- each connection uses a separate source token env var.

Example claims:

- `civil-and-program-records-match`;
- `eligible-for-combined-support`.
- `eligible-for-health-linked-support`.

### static-metadata-publisher

Functional role:

- serve generated registry metadata without running Relay;
- expose catalog, evidence-offering, policy, and profile artifacts as static
  files;
- demonstrate that metadata publication is a separate capability from runtime
  source APIs and evidence computation.

Generated bundle requirements:

- produced by `registry-manifest-cli` or the repository wrapper script;
- includes static evidence-offering documents;
- includes static policy metadata;
- includes DCAT/BRegDCAT artifacts where generated by the metadata tool;
- does not include runtime source paths, table ids, API key scopes, or source
  data.

Expected static paths:

```text
/metadata/index.json
/metadata/catalog.json
/metadata/evidence-offerings.json
/metadata/policies.jsonld
```

Per-offering and per-policy paths should follow the actual
`registry-manifest-cli publish` output layout.

## Demo Client

Functional role:

- run the narrated discovery and verification flow.

Implementation preference:

- Python standard library, following the existing narrated demo style.
- Reuse the existing demo helpers where practical:
  `demo/scripts/generate_demo_keys.py` for hashed key generation patterns and
  `demo/scripts/evidence_server_demo.py` for narrated HTTP flow patterns.
- Do not duplicate key hashing logic when it can be imported or factored into a
  small shared helper.

The demo client must not mount any `data/` directory. It may receive only the
client-facing credentials required by the flow, plus a row-reader credential for
one explicit positive row-read check.

Responsibilities:

- wait for all three Relay `/ready` endpoints;
- wait for all three Evidence Server `/.well-known/evidence-service` endpoints;
- fetch registry metadata;
- fetch the static metadata publication bundle;
- list evidence offerings;
- list static policy metadata;
- follow advertised Evidence Server discovery URLs;
- fetch Relay OpenAPI documents;
- fetch Evidence Server OpenAPI documents;
- call `/claims`;
- call `/claims/evaluate`;
- issue one demo credential from one Evidence Server;
- preserve and display one correlation ID across demo-client, Evidence Server,
  and Relay source calls where supported;
- run one batch evaluation;
- prove evidence-only credentials cannot read raw registry rows;
- include `Data-Purpose` for row reads and Evidence Server source calls;
- write full JSON artifacts under `demo/decentralized/output/`.

## Auth And Audit

Use API key or bearer-token auth everywhere.

OIDC is out of scope for v1. The demo must keep auth simple and explicit so the
main story remains registry decentralization, metadata discovery, Evidence
Server source access, scope separation, and audit. OIDC may be added later as an
optional Compose overlay.

All demo credentials must be generated local secrets:

- commit `.env.example`;
- do not commit `.env`;
- generate `.env` with `scripts/generate-demo-secrets.py`;
- mark generated values as local demo credentials in script output and README;
- do not print raw secrets in ordinary smoke or demo logs.

Required credential classes:

- metadata client;
- evidence client;
- row source token used by Evidence Server when calling Relay;
- row reader for one explicit positive row-read check;
- admin token only if the demo needs admin reload or diagnostics.

Evidence-only credentials must not be able to read Relay source rows.

### Relay Token Shape

Relay configs must store API key hash env var names, never raw API keys:

```yaml
auth:
  mode: api_key
  api_keys:
    - id: metadata_client
      hash_env: CIVIL_METADATA_CLIENT_HASH
      scopes:
        - civil_registry:metadata

    - id: evidence_source
      hash_env: CIVIL_EVIDENCE_SOURCE_HASH
      scopes:
        - civil_registry:metadata
        - civil_registry:evidence_verification

    - id: row_reader
      hash_env: CIVIL_ROW_READER_HASH
      scopes:
        - civil_registry:metadata
        - civil_registry:rows
```

The generated `.env` file provides the matching raw and hashed values:

```bash
CIVIL_METADATA_CLIENT_RAW=...
CIVIL_METADATA_CLIENT_HASH=sha256:...
CIVIL_EVIDENCE_SOURCE_RAW=...
CIVIL_EVIDENCE_SOURCE_HASH=sha256:...
CIVIL_ROW_READER_RAW=...
CIVIL_ROW_READER_HASH=sha256:...
```

Relay receives raw credentials in requests:

```http
Authorization: Bearer <raw-demo-token>
```

Relay validates the credential against the configured SHA-256 hash.

### Evidence Server Token Shape

Evidence Server configs must reference raw token env var names, never inline raw
values:

```yaml
auth:
  api_keys:
    - id: evidence_client
      token_env: CIVIL_EVIDENCE_CLIENT_TOKEN
      scopes:
        - civil_registry:evidence_verification
  bearer_tokens:
    - id: evidence_client
      token_env: CIVIL_EVIDENCE_CLIENT_BEARER
      scopes:
        - civil_registry:evidence_verification

evidence:
  source_connections:
    civil:
      base_url: http://civil-registry-relay:8080
      token_env: CIVIL_EVIDENCE_SOURCE_RAW
```

The Evidence Server source token must be the raw token whose hash is configured
in the corresponding Relay service. This proves the real flow:

```text
demo-client -> Evidence Server with evidence-client credential
Evidence Server -> Relay with evidence-source credential
Relay -> source-backed entity or DCI API
```

### Recommended Env Names

Per Relay authority:

```text
<DOMAIN>_METADATA_CLIENT_RAW
<DOMAIN>_METADATA_CLIENT_HASH
<DOMAIN>_EVIDENCE_SOURCE_RAW
<DOMAIN>_EVIDENCE_SOURCE_HASH
<DOMAIN>_ROW_READER_RAW
<DOMAIN>_ROW_READER_HASH
```

Per Evidence Server:

```text
<SERVER>_EVIDENCE_CLIENT_TOKEN
<SERVER>_EVIDENCE_CLIENT_BEARER
```

Shared Evidence Server:

```text
SHARED_EVIDENCE_CLIENT_TOKEN
SHARED_EVIDENCE_CLIENT_BEARER
SHARED_CIVIL_EVIDENCE_SOURCE_RAW
SHARED_SOCIAL_EVIDENCE_SOURCE_RAW
SHARED_HEALTH_EVIDENCE_SOURCE_RAW
```

Shared Evidence Server source connections must not reuse one source token
across Relay authorities.

### Audit And Observability

Use stdout audit for v1. Do not mix stdout and file audit by service.

Set tracing/logging consistently:

- `RUST_LOG=info`;
- JSON log format where supported by the binary;
- include request or trace IDs if already supported by middleware;
- demo-client must send one correlation header on every HTTP request, using the
  existing request-id or trace header supported by the services where possible;
- if a service does not propagate that header yet, the demo must still send it
  and record the limitation in the README;
- demo scripts must print the artifact path for every request they save.

Relay audit events must show:

- metadata reads;
- denied row reads;
- source API reads from Evidence Server;
- stable error codes for denied or invalid requests.

Evidence Server audit events must show:

- claim listing;
- successful evaluation;
- failed evaluation;
- source lookup outcome;
- redacted claim hashes or identifiers, without raw source records.

Audit write failures must remain fail-closed according to Evidence Server
behavior.

## Metadata Requirements

Each Relay metadata manifest must advertise:

- catalog identity;
- dataset identity;
- entity schemas;
- evidence offerings;
- policy metadata where relevant;
- Evidence Server endpoint and discovery URL for external offerings.

Static metadata publication is required in v1:

- generate a static bundle before or during Compose startup;
- serve the bundle from `static-metadata-publisher`;
- include evidence offerings and policy metadata;
- the demo client must discover at least one Evidence Server endpoint from the
  static metadata bundle, not only from Relay-hosted metadata.

For external Evidence Server offerings:

- `access.kind` must be `evidence-server`;
- Relay metadata may advertise endpoint and discovery URLs;
- Relay must not execute the offering through
  `POST /evidence-offerings/{offering_id}/verifications`.

Relay-native offerings are optional in this demo. If included, they must be
clearly labeled as Relay-native and must not be confused with Evidence Server
claim evaluation.

## Data Requirements

Fixture data must be:

- synthetic;
- small enough to inspect manually;
- deterministic;
- generated by script when practical;
- free of real personal data.

Minimum fixture coverage:

- one successful subject per registry;
- one missing subject;
- one subject that fails a derived predicate;
- one subject used in a cross-source claim;
- one child or dependent subject used for a support-eligibility scenario;
- one household whose benefit amount or eligibility changes when a member is
  deceased;
- one applicant whose eligibility depends on civil, social protection, and
  health facts.

Data format coverage for v1:

- civil registry: CSV;
- social protection registry: XLSX;
- health registry: Parquet preferred, documented XLSX fallback only if blocked.

If Parquet fixture generation becomes a blocker, keep the health registry in v1
with XLSX and record the Parquet blocker in the README.

The fixture generator must either produce all files or document any checked-in
binary fixture source of truth.

## Demo Scenarios

The demo should borrow the effective storytelling pattern from DPG
interoperability walkthroughs: a human journey row, a "what this means for the
person or family" row, and a system/API row. Unlike the slideware pattern, each
system/API step must map to a real Compose endpoint, saved artifact, policy,
audit event, and, where applicable, a credential.

Use functional domain names in v1. Do not label scenarios as OpenCRVS, OpenSPP,
OpenIMIS, MOSIP, OpenG2P, or similar real product flows until real product
instances or reviewed adapters are wired.

### Scenario 1: Birth Registration To Child Support

Purpose:

- show DCI-backed evidence verification;
- show Evidence Server as the right path when the caller should not see raw
  civil registry rows;
- show credential issuance from an Evidence Server result.

Human journey:

1. Birth or dependent child record exists in the civil registry extract.
2. A caregiver applies for child support.
3. The program needs to confirm birth date and alive status.
4. Eligibility is checked without exposing the civil registry row to the
   program client.
5. The result is rendered and a demo credential is issued.

System/API lane:

```text
demo-client
-> civil-registry-relay /metadata/evidence-offerings
-> civil-evidence-server /.well-known/evidence-service
-> civil-evidence-server /claims/evaluate
-> civil-registry-relay /dci/{registry}/registry/sync/search
-> civil-evidence-server /evidence/render
-> civil-evidence-server /credentials/issue
```

Required implementation notes:

- `civil-registry-relay` must expose the DCI route if this scenario uses
  `connector: dci`.
- the Relay demo image must therefore include `spdci-api-standards`;
- the demo client must attempt a civil row read with an evidence-only
  credential and receive HTTP 403;
- the successful evaluation must not include the full civil source row;
- the issued credential must be marked demo-grade.

Artifacts:

- Relay-hosted evidence offering;
- Evidence Server discovery document;
- OpenAPI documents;
- civil claim evaluation;
- row-denial problem response;
- CCCEV render;
- demo credential;
- Relay and Evidence Server audit records with correlation ID where supported.

### Scenario 2: Household Benefit Review From Registry Data

Purpose:

- show direct registry data consultation as a first-class lane after the split;
- show that Relay remains a protected data consultation gateway;
- show both filtered entity row reads and aggregate consultation with purpose,
  scopes, and audit.

Human journey:

1. A household benefit case is reviewed.
2. The program checks household membership or enrollment status.
3. A death or inactive member status changes the household situation.
4. The program recalculates a support decision outside Relay.
5. The family receives a decision artifact, not a Relay write-back.

System/API lane:

```text
demo-client
-> social-protection-registry-relay /metadata
-> social-protection-registry-relay /datasets/{dataset_id}/{entity}
-> social-protection-registry-relay /datasets/{dataset_id}/{entity}/aggregates
-> civil-evidence-server /claims/evaluate
-> civil-registry-relay source API
```

Required implementation notes:

- this scenario must include at least one direct Relay row read with a
  row-reader token and `Data-Purpose`;
- it must include at least one aggregate read with an aggregate-authorized
  token;
- it must also show an evidence-only credential failing to read the same row;
- it must show a row-reader-only credential failing to run the aggregate if
  aggregate scope is separated in the scenario config;
- any benefit "update" is represented as a demo decision artifact, not a write
  to Relay;
- the scenario should use Registry Data API for at least one Evidence Server
  source lookup so the demo is not DCI-only.

Artifacts:

- filtered household or enrollment row response;
- aggregate response;
- row-denial problem response;
- aggregate-denial problem response when aggregate scope is separated;
- recalculation decision JSON generated by the demo client;
- audit records for metadata, rows, aggregates, and denied access.

### Scenario 3: Cross-Authority Conditional Health Support

Purpose:

- show static metadata publication outside Relay;
- show cross-authority evidence composition;
- show cross-source CEL;
- show the health registry source format and separate source token.

Human journey:

1. An applicant applies for a conditional health or support package.
2. The client discovers the offering and policy from static metadata.
3. Civil facts confirm the applicant identity/status.
4. Social protection facts confirm household vulnerability or program status.
5. Health registry facts confirm local service availability or facility
   eligibility.
6. A shared Evidence Server derives the final eligibility decision.

System/API lane:

```text
demo-client
-> static-metadata-publisher /metadata/index.json
-> static-metadata-publisher /metadata/policies.jsonld
-> shared-eligibility-evidence-server /.well-known/evidence-service
-> shared-eligibility-evidence-server /claims/evaluate
-> civil-registry-relay source API
-> social-protection-registry-relay source API
-> health-registry-relay source API
```

Required implementation notes:

- the static metadata bundle must be generated from portable metadata, not
  scraped from a running Relay;
- the demo client must fetch policy metadata before evaluation;
- the shared Evidence Server must use distinct `token_env` values for each
  source connection it uses;
- the derived CEL claim must depend on claims backed by at least two Relay
  authorities;
- the health registry should use Parquet unless blocked and documented.

Artifacts:

- static metadata index;
- static evidence offering;
- static policy metadata;
- shared Evidence Server discovery document;
- cross-source evaluation result;
- missing-subject or failed-eligibility result;
- audit/log evidence that at least two Relay source connections were exercised.

The three scenarios together must cover:

- static metadata discovery;

## Demo Flow

`demo-flow.py` must perform and print a narrated flow:

1. Discover registry metadata from each Relay.
2. List datasets from each Relay.
3. List evidence offerings from each Relay.
4. Fetch static metadata from `static-metadata-publisher`.
5. List static policy metadata.
6. Extract advertised Evidence Server discovery URLs from both Relay-hosted and
   static metadata.
7. Fetch `/.well-known/evidence-service` from each Evidence Server.
8. Fetch Relay OpenAPI documents.
9. Fetch Evidence Server OpenAPI documents.
10. Fetch `/claims` from each Evidence Server.
11. Prove evidence-only credentials cannot read rows from at least one Relay.
12. Perform one positive row-read check with a row-reader token and
   `Data-Purpose`.
13. Perform one positive aggregate consultation with an aggregate-authorized
    token.
14. Evaluate one civil claim successfully.
15. Evaluate one social protection claim successfully.
16. Evaluate one health claim successfully.
17. Evaluate one shared cross-source CEL claim successfully.
18. Evaluate a missing-subject failure and save the artifact.
19. Run one batch evaluation with mixed outcomes.
20. Render CCCEV JSON-LD for one evaluation.
21. Issue one demo credential from one Evidence Server.
22. Run all three named demo scenarios and write a scenario summary artifact.
23. Save all responses as numbered JSON artifacts.

Credential issuance is required but demo-grade. It may use the existing static
demo issuer key path, must clearly label itself as demo-grade, and must not
claim production key management, revocation, status lists, or verifier metadata.

## Risks And Mitigations

| Risk | Mitigation |
| --- | --- |
| Demo accidentally teaches a database-backed Relay model | Do not add DB containers in v1. Use mounted CSV, XLSX, and Parquet source files. |
| DCI routes are missing from the Relay image | Use `Dockerfile.demo` and build with `spdci-api-standards` when configs use `connector: dci`. |
| Demo overclaims product integrations | Use functional names only. Add product names only after real adapters or product instances are wired. |
| Metadata and source API boundaries blur | Keep portable metadata free of runtime source paths and scopes. Only advertise standards-facing Evidence Server endpoints. |
| Evidence Server silently reads local fixtures | Mount no source data into Evidence Server containers. Source access must be HTTP to Relay services. |
| Demo client bypasses the architecture by reading fixtures | Do not mount `data/` into `demo-client`. |
| Cross-source CEL is under-tested | Add `decentralized_cross_source_cel` in Evidence Server and make shared Evidence Server exercise two Relay sources. |
| Auth is weakened for demo convenience | Require API key or bearer auth for protected routes. Prove row denial in the demo flow. |
| OIDC distracts from the decentralization story | Keep OIDC out of v1. Add it later as an optional Compose overlay only. |
| Raw tokens are accidentally committed | Generate `.env`, commit only `.env.example`, gitignore `.env`, and keep configs on env var references. |
| Compose service URLs leak into host-facing docs incorrectly | Document both host ports and internal Compose DNS URLs. |
| Future bitrot in image builds | Use the repo Rust toolchain or pinned base image already used by the project. Do not add floating ad hoc builder images. |
| Parquet fixture generation slows v1 | Keep the third health registry, use a documented XLSX fallback only if Parquet generation is the blocker. |

metadata. It should make the `registry-manifest` split obvious.

Recommended v1 improvements:

  from a running Relay process;
- include `index.json`, catalog JSON, evidence offerings, policy metadata,
  DCAT/BRegDCAT outputs when available, and entity JSON Schemas;
- expose a simple static index page or JSON index that links to the available
  metadata artifacts;
- include one evidence offering whose advertised Evidence Server endpoint is
  consumed by the demo client;
- include one policy artifact that the demo client fetches before evaluation,
  showing purpose or usage constraints beside the offering;
- keep static metadata free of source file paths, table ids, API key scopes,
  cache paths, or Relay runtime backend details;
- add one smoke check that compares a Relay-hosted offering and a
  static-published offering for the same functional domain, proving they can
  lead to the same Evidence Server without sharing a runtime;
- make the README explain the three publication modes:
  Relay-hosted metadata, static metadata publication, and Evidence Server
  discovery.

  running Relay at all;
