# Registry Platform: Pre-Beta Security Audit

- **Date:** 2026-05-30
- **Target:** first BETA tag line of `registry-platform` (`v0.1.2`; never been stable). A **shared security/operational library** (11 crates, ~10.5k LOC of Rust, each crate a single `lib.rs`) consumed by **Registry Relay** and **Registry Notary** by pinned git tag. It centralizes crypto primitives, audit chaining, OIDC verification, SD-JWT issuance, OID4VCI proof validation, replay/nonce stores, browser-facing HTTP security, and an outbound-HTTP/SSRF policy.
- **Scope:** the 11 `registry-platform-*` crates (`audit, authcommon, cache, crypto, httpsec, httputil, oid4vci, oidc, replay, sdjwt, testing`), `deny.toml`, `Cargo.toml`/`Cargo.lock`, and supply-chain posture. HEAD at audit time: `v0.1.2-13-gc549768`.
- **Threat model (library, not a server):** there is **no shipped server config** to audit. Every finding is judged by what a consumer gets from the public API, and classified by exposure:
  - **UNSAFE DEFAULT** â€” a consumer gets the insecure behavior just by using the public API with defaults (weighted heavily).
  - **FOOTGUN** â€” safe only if the caller opts in correctly; easy to get wrong (weighted).
  - **MISUSE-ONLY** â€” only exploitable if the caller does something the API does not invite (low weight).
- **Method:** orchestrated as a **workflow**, not single-threaded. 10 parallel finder lanes (one per security-relevant crate + a supply/hygiene lane) read the code and cited `file:line`; a cross-check lane traced the known relay/notary findings to their platform root; every finding was adversarially verified against the actual code, with a **2-lens panel for High/Critical** (lens A proves a consumer reaches it on defaults; lens B assumes it is already mitigated and hunts the guard); two completeness critics swept for missed surface and wrongly-dismissed issues, and their candidates were verified too. **59 agents, ~1.79M tokens, 615 tool calls.** Deterministic scans: `cargo audit`, `gitleaks`.
  - **Verification-provenance caveat (honest):** ~22 verification sub-agents failed to emit their structured verdict (a tool-call flake), so a slice of the per-finding panel results dropped out of the machine output. The **lead therefore hand-read `crypto`, `audit`, `replay`, `cache`, and the load-bearing sections of `oidc`, `oid4vci`, `sdjwt`, `httpsec`, `httputil`, `authcommon`, and `testing` directly**, and personally re-confirmed every Medium and a sample of the Low/Info findings against the cited lines. Findings below reflect the lead's adjudicated severity/exposure, which in several cases differs from the finder's or the panel's first pass (differences are called out).

## Verdict: CONDITIONAL GO for beta

**No critical vulnerabilities, and no clean High.** The crown-jewel surfaces are genuinely hardened: signing is **EdDSA-pinned** with ES/RS/PS rejected at parse (`crypto`), Ed25519 verification uses `verify_strict` (rejects malleable signatures), private-key seeds are zeroized and `Debug`-redacted with **no key-generation/silent-fallback primitive in the crate at all**, the OIDC verifier has **no insecure default algorithm allowlist** (`allowed_algorithms` is required and fails closed when empty, alg-confusion and `alg:none` are blocked, JWKS fetches route through the SSRF guard with DNS-pinning), the replay primitives are **atomic single-use** with collision-resistant hashed keys, the OID4VCI/SD-JWT holder-proof paths pin alg/`typ` and reject `jku/x5u/x5c/crit`, and the outbound-fetch *secure* path closes DNS-rebinding by pinning to validated socket addresses.

The blockers are **not** auth bypass, key disclosure, or forgery on defaults; they are a cluster of **platform-root unsafe-defaults and footguns** that the two consuming servers inherit, plus two gaps where the library that *owns* a security concern (SD-JWT presentation verification; OID4VCI proof freshness/replay) ships only half of it and silently delegates the dangerous half to every caller. For a shared security library the bar is: "the obvious API call is the safe one, and the dangerous path is named/typed so a caller cannot reach it by accident." Several primitives miss that bar.

## Medium findings (detail)

### M1 â€” Audit chain is unkeyed SHA-256; no keyed-chain mode (UNSAFE DEFAULT, platform root of relay-H4 / notary-M4)
**`crates/registry-platform-audit/src/lib.rs:64-65, 802-824, 873-878, 426-431, 628-697`** Â· CWE-345 / CWE-353

`record_hash` (`:802-824`) is `serde_json::to_vec(HashInput)` â†’ `sha256_bytes` (`:873-878`) â€” a **plain keyless SHA-256** over `{envelope_id, timestamp_unix_ms, prev_hash, record}`. No secret is an input. The per-deployment `AuditHashSecret`/`AuditKeyHasher` is consumed **only** at `:426-431` for `redact::QueryRedactor` ID/value pseudonymization, never by the chain. `verify_chain`/`verify_jsonl_lines` (`:628-705`) recompute the identical unkeyed hash. The crate's own test `verify_chain_with_trusted_tail_rejects_full_rewrite` (`:1338-1364`) asserts `verify_chain(&rewritten).is_ok()` for a fully fabricated chain.

**Attack through a consuming server:** relay/notary writes its audit log with `ChainState` + `JsonlFileSink` and later proves integrity with the default `verify_chain`. An actor with **write access to the JSONL file(s)** (compromised service account, log-volume mount, backup, lateral movement) deletes incriminating records, fabricates replacements, and recomputes a fully self-consistent chain using the open-source formula â€” no secret needed. `verify_chain` returns `Ok`. For a notary, a forged-clean trail can hide unauthorized issuance.

**Exposure/severity:** the default *and only* chain hash is unkeyed, so this is an **UNSAFE DEFAULT**; severity **Medium**, not High, because (a) it is post-compromise (requires audit-file write access) and (b) the crate ships a *real* mitigation â€” `verify_chain_with_anchors` / `ChainVerificationAnchors` with a `trusted_last_hash` stored off-host (`:578-648`), whose tests prove a full rewrite is rejected (`:1338-1391`). The defect is that the **secure path is opt-in while the insecure path is the default**, and there is **no HMAC-keyed chain option at all**.

**Fix (belongs HERE):** either (a) thread the existing `AuditHashSecret` into `record_hash` as HMAC-SHA256 when keyed (so a file-writer without the off-host secret cannot recompute a valid chain), or (b) make the anchored path the documented, ergonomic default for any persistent deployment. Relay-H4 and notary-M4 are symptoms of this keyless root; per `SECURITY.md` the fix lands here first, then propagates by tag, with a consumer-side anchoring mitigation in the interim.

### M2 â€” Default in-memory `insert_once` store is unbounded (UNSAFE DEFAULT, DoS)
**`crates/registry-platform-replay/src/lib.rs:351-355, 388-408`, `crates/registry-platform-cache/src/lib.rs:119-126, 149-183, 237-264`** Â· CWE-770

`InMemoryReplayStore::new()`/`default()` builds an `InMemoryCacheStore::new()` with `max_entries: None` (`cache:121-126`). `insert_once` stores one HashMap entry per distinct `jti` and **never auto-purges**: expired entries are removed only lazily when the *same* key is touched again (`cache:249-254`) or when a caller manually calls `purge_expired` (no timer/background task exists anywhere). There is **no `with_max_entries` on `InMemoryReplayStore`** (unlike `InMemoryConsumableNonceStore`, which defaults to a 4096 cap at `replay:388,399`), so a consumer cannot bound it without dropping to the lower-level `CacheReplayStore` + `InMemoryCacheStore::with_max_entries` by hand.

**Attack:** a consumer using the default store for `jti` one-time-use faces a stream of unique, far-future-expiry `jti`s (normal protocol traffic carries a fresh `jti` per request); the map grows without bound until OOM. The **asymmetry is the sharp edge**: the bounded nonce path is capped, but the harder-to-bound `insert_once` path is left uncapped by default.

**Fix (HERE):** give `InMemoryReplayStore` a `with_max_entries` constructor with a sane default cap (parity with the nonce store), and/or a periodic `purge_expired` sweep. At minimum make the two in-memory stores consistent.

### M3 â€” In-memory replay is the ergonomic default; Redis is an opt-in feature (FOOTGUN, notary-H2 enabler)
**`crates/registry-platform-replay/src/lib.rs:339-374, 388-425`, `crates/registry-platform-replay/Cargo.toml` (`default = []`, `redis = [...]`), `crates/registry-platform-cache/Cargo.toml` (`default = []`)**

The library has **no auto-selecting default store** (the consumer constructs one explicitly â€” good), and the in-memory store carries an honest non-durability warning (`:339-342`). But two things make the non-durable store the path of least resistance: (1) it exposes `Default` + a zero-arg `new()` needing no URL/connection/config, and (2) `RedisReplayStore` is behind `#[cfg(feature = "redis")]` with `default = []` in **both** `replay` and `cache`, so a consumer who depends on the crate without enabling the feature **cannot even construct the durable backend** â€” the only store that compiles in is the ephemeral one. One-time-use then silently becomes non-durable on restart and single-instance behind a load balancer (the `insert_once` name implies a stronger guarantee than the default delivers).

**Cross-check nuance:** notary-H2 (the in-memory replay default) **roots partly here** (this crate makes the ephemeral store the easy path and gates durability behind a feature) but the *silent default selection* is consumer-side (notary's `config.rs` defaults `replay.storage = "in_memory"`). Notary *does* enable `features = ["redis"]`, so its durable backend exists; its H2 is a config default, not a platform compile barrier. **Fix: both** â€” platform: rename to make non-durability unmissable (e.g. `EphemeralSingleProcessReplayStore`) or require an explicit backend choice via a builder; consumer: refuse to start in production on in-memory.

### M4 â€” `expected_nonce: None` silently disables OID4VCI proof freshness (FOOTGUN)
**`crates/registry-platform-oid4vci/src/lib.rs:205-210, 276-287, 357-374`**

`ProofValidationPolicy.expected_nonce` is `Option<&str>` (`:207`). In `validate_proof_jwt`, when it is `None` the entire server-challenge block (`:280-284`) is skipped, so no issuer challenge is bound into the proof; `exp` is optional (`:286`, `validate_time` only checks `exp` when present), so a proof carrying just `iat` is accepted for the whole `max_lifetime` window. The doc comment (`:251-254`) states plainly that nonce handling/replay is a caller responsibility. `None` is a first-class, inviting value with no compile-time or runtime signal that freshness is now off; `CredentialIssuerMetadata.nonce_endpoint` is itself `Option`, reinforcing the "nonces are optional" reading.

**Attack:** an issuer (notary/relay) calls `validate_proof_jwt` with `expected_nonce: None` (the exact shape in the crate's own `validates_did_jwk_kid_proof` test); an attacker who observes one proof JWT replays the identical proof to the credential endpoint and is issued a fresh credential bound to the holder key, with no proof the holder is live now. PoP degrades from "holder is answering this issuer's challenge" to "someone once held this key."

**Fix (HERE):** make the challenge non-optional (`expected_nonce: &str`), or split into `validate_proof_jwt_with_nonce` vs an explicitly-named `..._without_nonce` so the insecure path screams at the call site; when `None`, require the proof to carry no nonce and return a distinct `ValidatedProof` state so callers cannot mistake an unchallenged proof for a challenged one.

### M5 â€” OID4VCI proof validation does no replay tracking and gives no type-level signal (FOOTGUN)
**`crates/registry-platform-oid4vci/src/lib.rs:251-254, 276-318, 598-611`**

Even with a correct `expected_nonce`, `validate_proof_jwt` only checks equality against the single expected value (`:280-284`); it does not record consumed nonces (the in-tree test `validate_proof_jwt_does_not_track_nonce_reuse_across_calls` proves the same proof validates twice). A successful `ValidatedProof` is byte-identical whether or not the caller wired a replay store, so the omission is invisible at the call site, and the crate ships no used-nonce store and no consumption token to force the obligation.

**Attack:** an issuer that reuses or fails to single-use-retire its `c_nonce` (a separate component the library does not provide) lets an attacker replay a captured proof to mint multiple credentials for one challenge.

**Fix (HERE):** ship a single-use nonce store trait/helper in this crate (or accept `&mut impl NonceConsumer` in the policy) so retiring the nonce is part of the validated path; or return a non-`Clone` consumption token the caller must redeem, making the obligation type-enforced. The `replay` crate already provides the atomic primitive â€” wire it in or reference it.

### M6 â€” No SD-JWT presentation verifier in the crate that owns SD-JWT (FOOTGUN)
**`crates/registry-platform-sdjwt/src/lib.rs:32-97, 190-255`** Â· CWE-347

The public surface is `SdJwtIssuer::{from_jwk, from_signing_provider, issue}`, `new_credential_id`, `sort_sd_digests`, and `validate_holder_proof`. **None verifies a presented SD-JWT.** There is no function that (1) verifies the issuer's EdDSA signature over the SD-JWT, (2) base64url-decodes each presented disclosure, SHA-256-hashes it, and confirms membership in `_sd`, and (3) rejects any disclosure whose digest is not in `_sd`. A consumer accepting SD-JWT presentations must implement digest recomputation itself, and a consumer that omits step (2)/(3) lets a holder append a forged disclosure the issuer never signed and have its claims trusted. The **selective-disclosure forgery defense is simply not provided by the library that owns SD-JWT.**

**Fix (HERE):** add a presentation-verification function (verify issuer JWT signature against the issuer public JWK, assert `_sd_alg == sha-256`, recompute `base64url(SHA-256(disclosure))` per presented disclosure, require each in `_sd`, reject extras, return resolved claims). This keeps forgery rejection inside the crate rather than re-implemented (possibly wrongly) by each consumer.

### M7 â€” Holder proof binds to a caller-supplied opaque `disclosure_hash`, not the presentation bytes (FOOTGUN)
**`crates/registry-platform-sdjwt/src/lib.rs:172-178, 190-255` (esp. `:214, 235-241`)** Â· CWE-345

`validate_holder_proof` verifies the KB-JWT signature against a **caller-supplied `holder_jwk`** (`:214`) and binds to `HolderProofBindings.disclosure_hash` (an opaque `&[u8]` the caller computes elsewhere, compared at `:235-236`) plus a caller-supplied `claim_set` (`:239`). The crate **never computes the SD-JWT `sd_hash`** (`SHA-256` over `<issuer-jwt>~<disclosures>~`) from the actual issuer JWT + presented disclosures. If a consuming notary computes `disclosure_hash` over the wrong material (claim values rather than the canonical presentation, or a stale buffer), the KB-JWT still "validates" while being bound to nothing the issuer signed. The signature check is real; the link from PoP to the specific issued credential is delegated wholesale to the caller with no documented derivation.

**Fix (HERE):** provide a presentation-verification function (see M6) that recomputes the canonical `sd_hash` and feeds it as the binding, or at minimum ship a constructor that computes `HolderProofBindings.disclosure_hash` the one correct way and document the exact byte derivation.

### M8 â€” Access-token verification skips `typ` binding when `allowed_typ` is empty (FOOTGUN)
**`crates/registry-platform-oidc/src/lib.rs:523, 601-647` (esp. `:610-618`)** Â· CWE-345

`verify_access_token` consults the `typ` allowlist **only when it is non-empty** (`:610`). `allowed_typ` is a plain `Vec<String>` (`:523`) with no default and no enforcement when empty (the crate's own `hs256_test_verifier` ships it empty). With an empty list, **any `typ` (or none) is accepted** on the access path; only alg/kid/iss/aud/exp/nbf/client are checked. An attacker holding an ID token (or any JWT minted by the same issuer/keys whose `aud` overlaps the resource audience) can replay it to the access-token entrypoint, because nothing pins the token to RFC 9068 `at+jwt`.

**Fix (HERE):** require a non-empty `allowed_typ` for the access path (reject construction or fail closed at verify if empty), or default the access path to require `at+jwt` unless the consumer explicitly opts out. The alg-pinning and aud/iss/exp enforcement around this are otherwise solid (see "What holds up").

### M9 â€” `did:jwk` parse does not enforce canonical encoding â†’ unstable `holder_id` (FOOTGUN)
**`crates/registry-platform-crypto/src/lib.rs:71, 390-405, 548-557`, `crates/registry-platform-oid4vci/src/lib.rs:292-308`** Â· CWE-1023

`parse_did_jwk` (`crypto:390-405`) base64url-decodes the embedded JSON and runs `PublicJwk::parse`, which has no `deny_unknown_fields` (`:71`) and only rejects the 7 private members (`:548-557`) â€” it never checks the encoding was canonical. So a holder can present `did:jwk:<B>` where `B` encodes a JWK with reordered keys, inserted whitespace, or extra ignored members; all decode to the **same** `PublicJwk` (so the `kid_jwk != holder_jwk` equality at `oid4vci:299` passes), but the **raw DID string differs every time**. `oid4vci` sets `holder_id = did.to_string()` (`oid4vci:302`) and only requires `iss == holder_id` (`:306`), which the attacker satisfies. One key thus yields unlimited distinct `holder_id` values.

**Attack:** any consumer that uses `ValidatedProof.holder_id` as a stable subject key (dedup, per-holder rate limit, one-credential-per-holder, revocation/allowlists) is bypassed â€” the same key looks like N different holders.

**Fix (HERE):** in `parse_did_jwk`, after `PublicJwk::parse`, re-encode via `did_jwk_from_public_jwk(&jwk)` and reject if the supplied identifier differs from the canonical one, making the `did:jwk` string a stable function of the key.

### M10 â€” `FetchUrlPolicy::validate()` discards the DNS pin â†’ rebinding bypass if used as the guard (FOOTGUN)
**`crates/registry-platform-httputil/src/lib.rs:204-213, 221-284, 362-379`** Â· CWE-918 / CWE-367

`validate()` (`:211-213`) calls `validate_for_immediate_fetch(url).map(|_| ())` â€” it resolves DNS, checks the IPs, then **throws away the `ValidatedFetchUrl`** (the resolved-address pin). The subsequent real request re-resolves the hostname. A caller who treats `validate()` as the SSRF guard and then fetches with a normal client is exposed to DNS rebinding: the attacker's resolver returns a public IP during `validate()` and `169.254.169.254` / `127.0.0.1` / `10.x` during the fetch. The doc comment warns it "must not be used as the sole guard," but the name and the `Result<(), _>` signature actively invite exactly that pattern.

**The secure path is solid** (see "What holds up"): `validate_for_immediate_fetch` returns the pin and `immediate_get` pins reqwest's resolver to the validated `resolved_addrs` (`:375`), closing the rebinding window; the in-tree consumer (`oidc`) uses it.

**Fix (HERE):** remove `validate()`, or `#[deprecated]` + rename to `validate_shape_only`, or make it return the `ValidatedFetchUrl` so the only ergonomic next step is `immediate_get()`. Make the secure path the path of least resistance.

### M11 â€” `testing` crate is not gated; private key material ships as a normal library (FOOTGUN)
