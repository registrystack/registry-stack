Use this tutorial to prove the combined Registry Relay and Registry Notary path without contacting
a registry or starting a deployment. The built-in HTTP project has one Relay integration, one
Notary evidence service, and synthetic fixtures for match, no-match, and ambiguity.

<QuickstartMeta
  outcome="A fixture-tested claim journey and separate unsigned Relay and Notary Config Bundle inputs."
  time="About 5 minutes"
  level="Local authoring with synthetic fixtures"
  prerequisites={['A registryctl build with project authoring commands', 'A shell']}
/>

Do not add production endpoints, credentials, or subject records to this exercise.

{/* Evidence: crates/registryctl/assets/project-starters/bounded-http and
    crates/registryctl/src/project_authoring/fixtures.rs. */}

## Copy the combined starter

Create a local Registry Stack project:

```sh
registryctl init --from http --project-dir registry-claim-project
```

The report contains:

```text
"status": "initialized"
```

The authored `registry-stack.yaml` defines one evidence service. Its consultation maps a bounded
request input to the Relay integration, and its claims derive only from the consultation outcome
and typed output.

## Evaluate the matching fixture

Run only the active-person fixture and include its safe synthetic trace:

```sh
registryctl test \
  --project-dir registry-claim-project \
  --integration person-record \
  --fixture active-person \
  --trace
```

The JSON report identifies the exercised integration, fixture, consultation outcome, output, and
claims. The important fields are:

```json
{
  "status": "passed",
  "fixtures": [
    {
      "integration": "person-record",
      "fixture": "active-person",
      "outputs": ["active"],
      "claims": ["person-active", "person-record-exists"],
      "outcome": "match",
      "passed": true
    }
  ]
}
```

The fixture runner executes the Relay acquisition contract in memory, then evaluates the Notary
claims from the typed consultation result. It does not permit a fixture to choose a destination,
credential, capability, worker command, or decoder.

## Verify no-match semantics

Run the no-match fixture:

```sh
registryctl test \
  --project-dir registry-claim-project \
  --integration person-record \
  --fixture no-person
```

The report contains `"outcome": "no_match"` and `"status": "passed"`. The fixture contract expects
the presence claim to be `false` and the direct output claim to be null. No source error or raw
record becomes a Notary claim.

Run every fixture to include the ambiguity case and derived security checks:

```sh
registryctl test --project-dir registry-claim-project
```

The report must again contain:

```text
"status": "passed"
```

## Inspect the authored policy

Open `registry-claim-project/registry-stack.yaml`. The evidence service owns:

- the purpose and legal basis;
- caller scopes;
- the consultation input mapping;
- claims and disclosure modes; and
- the credential profile claim list.

The integration owns the fixed source request and minimized typed output. Product and version
metadata document interoperability evidence. They do not select the HTTP or script capability.

Change the service `purpose`, then rerun the fixture test. Purpose policy is authored once and
compiled into both the Relay consultation profile and the Notary claim contract. Do not patch the
generated Notary YAML.

## Validate the cross-product contract

Compile the project through both product validators and print the redacted plan:

```sh
registryctl check \
  --project-dir registry-claim-project \
  --environment local \
  --explain
```

The report contains `"status": "valid"`. In the explanation, confirm that Notary names the same
consultation profile, contract hash, purpose, closed inputs, outcome union, and typed outputs that
Relay exposes. A mismatch fails validation rather than selecting a fallback source path.

## Build unsigned product inputs

Build the reviewed project:

```sh
registryctl build \
  --project-dir registry-claim-project \
  --environment local
```

The report contains `"status": "built"`. Inspect the generated inputs:

```sh
sed -n '1,220p' \
  registry-claim-project/.registry-stack/build/local/private/notary/config/notary.yaml
```

The Notary config has one logical Relay connection and one compiler-pinned consultation. It has no
source destination, source credential, product adapter, or script runtime. Those remain Relay
responsibilities.

The generated Relay and Notary directories are separate unsigned inputs to each product's Config
Bundle workflow. They are not a signed project-level deployment bundle and should not be committed.

## Clean up

Remove the generated project when you no longer need it:

```sh
rm -rf registry-claim-project
```
