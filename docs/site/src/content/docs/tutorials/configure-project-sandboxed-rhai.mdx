Use this guide only when a reviewed live HTTP recipe cannot be expressed as declarative
bounded HTTP operations.
You will keep every request and response shape in the integration, add one orchestration script,
and enable the advanced capability through an operator-security review.

## When to use this

Use bounded HTTP when a fixed dependency graph and declarative conditions express the journey.
Use `script` only when the reviewed recipe needs bounded choices among predeclared
operations or closed output assembly that would otherwise be hard to maintain.

The script is not a general connector.
It cannot define a destination, method, path, parameter, credential, decoder, response field,
policy, claim, disclosure mode, or provenance.

## Before you start

You need:

- A Registry Stack project whose ordinary
  [bounded HTTP authoring journey](../author-registry-project/) passes offline.
- One to five complete bounded HTTP operations with fixed request and closed response contracts.
- Synthetic fixtures for each script branch, denial case, malformed response, ambiguity, timeout,
  and resource-budget failure.
- An operator-security reviewer for the advanced capability and worker deployment.

Keep source product and version as review metadata only.
The authored `runtime: rhai_v1` contract selects the Rhai runtime independently of either value;
runtime availability is governed by the implementation-owned runtime and ABI capability table.

## Declare the reviewed operations

Replace `capability.http` with `capability.script`, select the reviewed `rhai_v1` runtime, add the script path, and retain
the complete fixed operation contracts:

```yaml
capability:
  script:
    runtime: rhai_v1
    script: orchestration.rhai
    credential: { type: static_bearer }
    operations:
      identity:
        request:
          method: GET
          destination: data
          path: /v1/people
          query:
            personId: { input: person_id }
            limit: { value: 2 }
        response:
          statuses: [200]
          max_bytes: 32768
          schema:
            type: object
            additional_fields: reject
            fields:
              records:
                required: true
                type: array
                max_items: 2
                items:
                  type: object
                  additional_fields: reject
                  fields:
                    status: { required: true, type: string, max_bytes: 32 }
          cardinality: { records: records, mode: probe_two }
```

Declare `person_id` as the one typed integration input and declare each terminal output in the
integration's closed `outputs` block.
Add any dependent operation beside `identity` with its complete fixed request, response, and
cardinality contract.

The script can choose only names from this compiled operation set.
It cannot construct a request or introduce another source call.

## Add the orchestration script

Create the declared `orchestration.rhai` file with the fixed `consult(input, prior)` entry point.
This example chooses one declared operation, handles no match, and returns the closed outputs:

```txt
fn consult(input, prior) {
    if !prior.contains("identity") {
        return #{ operations: ["identity"], outputs: #{} };
    }
    if !prior.identity.presence {
        return #{ operations: [], outputs: #{
            exists: #{ type: "presence", value: false },
            status: #{ type: "string", value: () }
        }};
    }
    return #{ operations: [], outputs: #{
        exists: #{ type: "presence", value: true },
        status: #{ type: "string", value: prior.identity.status }
    }};
}
```

The worker receives only typed integration inputs, typed prior operation outputs, allowed
operation names, the closed output schema, and fixed resource limits.
It does not receive caller identity, scopes, purpose, service, claim, format, disclosure mode,
authorization material, credentials, destination authority, raw responses, or provenance.
The Rhai engine has no network, filesystem, process, environment, clock, or random interface.

Return either more declared operation names or the complete terminal output map.
An unknown operation, wrong output name or type, incomplete terminal map, duplicate choice, or
resource-budget breach fails closed.

## Enable the advanced capability

Add explicit operator-security enablement to the environment integration binding:

```yaml
integrations:
  person-record:
    source_version: project-contract-v1
    data_destination: { origin: https://project-source.invalid }
    advanced_capabilities:
      script:
        enabled: true
        review: operator_security
    credential:
      type: static_bearer
      token: { secret: PROJECT_SOURCE_BEARER_TOKEN }
      generation: 1
```

This opt-in is necessary but not sufficient.
The release capability table must admit the implementation-owned `rhai_v1` runtime,
and the generated Relay binding must preserve its compiled operation and resource limits.
No project-authored field can select a worker executable, worker path, worker digest, or worker
sandbox ceiling. Authored integration bounds still select the reviewed source-byte, timeout, and
concurrency limits within those compiler-owned ceilings.

For production activation, Registry Relay requires Linux and the code-owned
`registry-relay-rhai-worker` executable installed beside Relay.
Each evaluation runs in a fresh, environment-scrubbed, one-shot process with a maximum 128 MiB
address-space ceiling and the compiled CPU, instruction, depth, collection, output, IPC, wall-time,
and concurrency limits.
Treat a missing worker or unavailable sandbox as an unready or failed integration, not a reason to
fall back to in-process script execution.

## Verify the reviewed branches

Run the offline fixtures and semantic check:

```sh
registryctl test --project-dir <project>
registryctl check \
  --project-dir <project> \
  --environment <environment> \
  --explain
```

The reports contain `"status": "passed"` and `"status": "valid"`.
Confirm that the explanation requires `operator_security`, lists only the declared operation
names, and exposes fixed resource ceilings without script source, credential values, or private
source observations.

Fixture success proves the reviewed script against synthetic observations and the worker
contract.
It does not prove source interoperability, production sandbox availability, legal suitability, or
deployment approval.

## Troubleshooting

| Symptom | Cause | Fix |
| --- | --- | --- |
| `script requires a released authoring contract and explicit operator-security enablement` | The runtime release gate or environment opt-in is absent. | Use a release that admits the reviewed runtime and add the explicit operator-security enablement. |
| The script selects an unknown operation | The returned name is outside the compiled operation set. | Return only a predeclared operation name. |
| A terminal result is rejected | The script omitted an output or returned a name, type, nullability, or bound outside the closed schema. | Return the complete output map with the declared scalar types and bounds. |
| The integration is unready on a production host | Linux isolation or the adjacent code-owned worker is unavailable. | Correct the reviewed deployment; do not configure an alternate worker or in-process fallback. |
