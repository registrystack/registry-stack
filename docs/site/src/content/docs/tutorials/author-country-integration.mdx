Create a country-owned bounded HTTP integration without editing Registry Stack source code.
You will copy the built-in starter, prove its synthetic journey offline, inspect the bounded
acquisition and disclosure plan, and build separate unsigned inputs for Registry Relay and
Registry Notary.

<QuickstartMeta
  outcome="A tested country workspace with reviewable bounded HTTP configuration and unsigned Relay and Notary bundle inputs."
  time="About 10 minutes after registryctl is installed"
  level="Local authoring with synthetic fixtures"
  prerequisites={['A registryctl build with country authoring commands', 'A shell and a text editor']}
/>

This tutorial uses a fictional registry and synthetic records.
Do not add source credentials, production endpoints, or real subject records.

## Copy the bounded HTTP starter

Create a local country workspace:

```sh
registryctl init --from bounded-http --country-dir country-registry
```

The JSON report contains:

```text
"status": "initialized"
```

The command creates this authored layout:

```text
country-registry/
  registry-stack.yaml
  integrations/
    person-record/
      integration.yaml
      fixtures/
  environments/
    local.yaml
```

The starter is copied into your workspace.
Registry Stack does not resolve a remote package or download integration code at runtime.

## Prove the starter offline

Run every fixture before changing the source contract:

```sh
registryctl test --project country-registry
```

The JSON report contains:

```text
"status": "passed"
```

The fixture set covers match, no match, malformed and oversized responses, invalid input,
unexpected source status, timeout, wrong caller or scope, wrong purpose, and attempted request
authority changes.
The command does not contact Registry Relay, Registry Notary, or a source registry.

## Adapt the authored contract

For this synthetic journey, make these exact substitutions with your editor:

| File | Field | Replace | With |
| --- | --- | --- | --- |
| `registry-stack.yaml` | `registry.id` | `fictional-citizen-registry` | `fictional-civic-support-registry` |
| `integrations/person-record/integration.yaml` | `source.product` | `fictional-citizen-system` | `fictional-civic-support-api` |
| `integrations/person-record/integration.yaml` | `source.versions.unverified` | `[country-fixture-v1]` | `[tutorial-contract-v2]` |
| `integrations/person-record/integration.yaml` | the `person` request `path` | `/v1/person-records` | `/v2/citizens/status` |
| `environments/local.yaml` | `source_version` | `country-fixture-v1` | `tutorial-contract-v2` |
| `environments/local.yaml` | `data_destination.origin` | `https://citizen-registry.invalid` | `https://civic-support.invalid` |

These values remain fictional and `.invalid` destinations cannot identify a live service. The
fixture observations describe response bodies rather than executing HTTP requests, so they remain
valid after the fixed request paths change.

For an actual country integration, review all four authored surfaces and replace the fictional
contract with approved country values:

1. In `registry-stack.yaml`, name the registry, service, purpose, caller scopes, consultation
   input mapping, facts, claims, disclosure modes, and credential claim allow-list.
2. In `integrations/person-record/integration.yaml`, define the typed subject input, fixed
   operation, closed response schema, fact mapping, cardinality, bytes, calls, deadline, and
   concurrency.
3. In `integrations/person-record/fixtures/`, replace the synthetic source observations and
   expected facts and claims. Keep positive and denial fixtures.
4. In `environments/local.yaml`, bind the source version, non-production destination, secret
   references, caller fingerprints, Relay trust, and deployment services.

The integration fixes the destination slot, method, path, parameter names, response fields, and
bounds before deployment.
Consultation inputs can populate only declared value slots.
They cannot select an origin, operation, credential, codec, projection, or response mapper.

Use `source.product` as review metadata for the country system.
It does not select a product executor or runtime capability.
Mark a country version `unverified` until the required integration evidence exists.

## Rerun the fixture journey

Test the edited project:

```sh
registryctl test --project country-registry
```

The report must again contain:

```text
"status": "passed"
```

Failures identify the authored file and fixture.
A fixture can supply bounded source observations, but it cannot supply a destination, credential,
worker command, authentication bypass, or decoder mode.

## Review the generated plan

Compile the authored project through the Relay and Notary validators without writing build
output:

```sh
registryctl check \
  --project country-registry \
  --environment local \
  --explain
```

The JSON report contains:

```text
"status": "valid"
"baseline": "initial_without_baseline"
```

Review the fixed operations, input mappings, caller scopes, purpose, facts, claims, disclosure
modes, and credential membership in `explanation`.
An initial project without an approved baseline requires the `claim`, `integration`,
`country_policy`, and `operator_security` review classes.
A valid report is not legal approval, source interoperability evidence, or deployment authority.

## Build unsigned product inputs

Generate deterministic inputs for the existing Config Bundle v1 workflow:

```sh
registryctl build \
  --project country-registry \
  --environment local
```

The JSON report contains:

```text
"status": "built"
```

The output root contains:

```text
country-registry/.registry-stack/build/local/
  reviewable/
    integration-packs/
    consultation-contracts/
    review.json
  private/
    relay/
      config/relay.yaml
      approval/review.json
    notary/
      config/notary.yaml
      approval/review.json
```

The product input directories remain separate and contain secret references, not secret values.
`build` does not create a bundle manifest, sign a bundle, or read a secret value.
Treat every generated directory as institutionally sensitive and keep it out of source control.

## Clean up generated output

Keep `country-registry/` as the authored country project.
If you do not need the generated inputs, remove only
`country-registry/.registry-stack/build/local/`.
You can reproduce that directory with the same `build` command.

## What you built

You copied a local bounded HTTP starter, proved its positive and denial fixtures without source
access, adapted the reviewed country contract, inspected the complete redacted plan, and generated
separate unsigned Relay and Notary inputs.
The generated profile keeps source acquisition in Relay and claim disclosure in Notary.

## Troubleshooting

| Symptom | Cause | Fix |
| --- | --- | --- |
| `country project destination must be absent or an empty real directory` | `init` found content, a non-directory, or a symlink. | Choose an absent or empty real directory. |
| `input.pattern_mismatch` | A fixture input does not satisfy the declared type or pattern. | Correct the synthetic fixture or the reviewed input contract. |
| `fixture.request_override_forbidden` | A fixture attempted to choose request authority. | Keep transport and credential authority in the integration and environment. |
| `--against and --anchor must be supplied together` | Only one approved-baseline argument was present. | Supply the verified bundle and matching trust anchor together. |
| Generated configuration fails validation | Authored fields cannot produce a valid Relay or Notary startup contract. | Correct the reported authored file and field, then rerun `test` and `check`. |
